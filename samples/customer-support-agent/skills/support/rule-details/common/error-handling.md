# Error Handling — エラーシナリオとリカバリ手順

## このファイルの目的

カスタマーサポートエージェントが遭遇しうる6つのエラーシナリオに対する検出条件・影響範囲・対応手順・エスカレーション判断を定義する。
**Domain Research の4 Pitfalls**（ハルシネーション / PII漏洩 / エスカレーション判定ミス / トーンミスマッチ）への直接対策を含む。

---

## エラー対応の基本原則

1. **フェイルセーフ**: エラー発生時は安全な方向（エスカレーション・保留）へ倒す
2. **ログ必須**: すべてのエラーは監査ログに記録する（ISO 8601 タイムスタンプ付き）
3. **顧客通知**: 対応遅延が発生する場合は暫定回答を送信する
4. **上限管理**: リペアパスは最大1回、総リトライは最大3回

---

## シナリオ 1: KB未ヒット

**対応するPitfall**: Pitfall 1（ハルシネーション回答）

### 検出条件
- KB信頼度スコア < 0.4
- KB検索がゼロ件ヒット

### 影響
- Critical — KBにない情報を「それらしく」生成するとハルシネーション回答が顧客に届く

### 対応手順

```
Step 1: KB検索を Keyword → Semantic → Category の順で再試行（1回のみ）
Step 2: 再試行後もスコア < 0.4 の場合 → エスカレーションモードへ切り替え（R4）
Step 3: 顧客への暫定回答を生成する（以下テンプレート使用）
Step 4: 引き継ぎサマリーに「KB未ヒット」を明記してオペレーターへ渡す
Step 5: 監査ログにエラー記録
```

### 暫定回答テンプレート（KB未ヒット）

```
{顧客名} 様

お問い合わせいただきありがとうございます。
いただいたご質問について、より正確な情報をご提供するため、
専門の担当者が確認の上、改めてご連絡いたします。

通常 {SLA目標時間} 以内にご回答いたします。
大変お待たせして申し訳ございません。

{署名}
```

### GOOD / BAD

**GOOD**: KB信頼度スコアが 0.3 だったため、エスカレーションモードに切り替え、「確認後ご連絡します」テンプレートを送信した。

**BAD**: KB未ヒットにもかかわらず、「おそらく設定メニューから変更できます」と推測で回答した。
→ 理由: Domain Research Pitfall 1（ハルシネーション）。存在しない手順を案内する重大リスク。

## シナリオ 2: PII漏洩検出

**対応するPitfall**: Pitfall 2（PII漏洩）

### 検出条件
- Q1 PIIスキャンで未マスキングのPIIが出力ドラフトに検出された
- Q2 再スキャンでPIIが残存している

### 影響
- Critical — 法的リスク、GDPR/個人情報保護法違反、顧客信頼の失墜

### 対応手順

```
Step 1: 出力ドラフトの送信を即時停止
Step 2: content-validation.md のマスキングルールを再適用
Step 3: 全PIIパターンで再スキャン実施
Step 4: 誤検知除外ルールを確認（製品名・URL等の除外確認）
Step 5: 再スキャンで「clean」を確認してから送信
Step 6: 監査ログに検出箇所・対応内容を記録
Step 7: 同一チケットで2回以上PII漏洩が検出された場合 → マネージャーへ通知
```

### 緊急停止条件

Q1を2回通過してもPIIが残存する場合 → 送信を完全停止し、人間レビュー待ちステータスに移行する。自動送信は禁止。

### GOOD / BAD

**GOOD**: Q1でクレジットカード番号を検出し、`****-****-****-3456` にマスキング後、再スキャンで「clean」を確認してから送信した。

**BAD**: SLAが迫っているため、Q1をスキップして回答を送信した。
→ 理由: Q1はいかなる条件でもスキップ禁止（`core-workflow.md` Q1: NEVER SKIP）。

---

## シナリオ 3: エスカレーション先不在

**対応するPitfall**: Pitfall 3（エスカレーション判定ミス）

### 検出条件
- R4 でエスカレーション先チームが営業時間外
- 担当チームのキャパシティ超過
- エスカレーション先が存在しないカテゴリ

### 影響
- High — エスカレーションが完了せず顧客対応が宙吊りになる

### 対応手順

```
Step 1: 次点のエスカレーション先を確認（カテゴリ別フォールバックチーム）
Step 2: フォールバック先も不在の場合 → マネージャーへ緊急通知
Step 3: チケットを「Escalated（待機）」ステータスに設定
Step 4: 顧客への暫定回答を送信（対応遅延を正直に伝える）
Step 5: SLA超過リスクがある場合 → SLA警告フラグを立て、マネージャーに即通知
Step 6: 監査ログに不在理由・フォールバック試行記録を残す
```

### カテゴリ別フォールバックチーム

| 一次チーム | フォールバック先 |
|-----------|--------------|
| セキュリティチーム | セキュリティマネージャー（24時間対応） |
| 課金チーム | 一般サポートマネージャー |
| テクニカルサポート | シニアサポートエージェント |
| カスタマーサクセス | サポートマネージャー |

### GOOD / BAD

**GOOD**: セキュリティチームが不在だったため、セキュリティマネージャーへ緊急エスカレーションし、顧客には「最優先で対応中」の暫定回答を送信した。

**BAD**: エスカレーション先が不在だったため、KB未確認のまま自己判断で回答を生成した。
→ 理由: Domain Research Pitfall 3（エスカレーション判定ミス）。エスカレーション必要案件を自律処理するリスク。

## シナリオ 4: SLA超過

### 検出条件
- `sla.sla_remaining_minutes <= 0`（SLA超過フラグ）
- `sla.sla_remaining_minutes <= 15`（SLA警告フラグ）

### 影響
- High — 顧客との契約上の義務違反、顧客不満、チャーンリスク

### 対応手順

```
【SLA警告（残 <= 15分）】
Step 1: Q2コンテンツレビューを簡略化（KB照合のみ、詳細事実確認スキップ）
Step 2: Q1（PIIスキャン）は省略しない（必ず実施）
Step 3: 処理中の担当者へ警告通知

【SLA超過（残 = 0）】
Step 1: SLA超過フラグをチケットに記録
Step 2: マネージャーへ即時通知
Step 3: 顧客への謝罪+対応中の暫定回答を送信
Step 4: 監査ログにSLA超過と理由を記録
Step 5: Q4ディスパッチ後のSLAレポートに超過件数として計上
```

### GOOD / BAD

**GOOD**: SLA残 10分の警告を受け、Q2を簡略化（KB照合のみ）しながらQ1は必ず実施し、SLA内に回答を送信した。

**BAD**: SLA残 5分のためQ1（PIIスキャン）もスキップして送信した。
→ 理由: Q1は SLA 超過でもスキップ禁止。PIIを含む回答が顧客に届く重大リスク。

## シナリオ 5: トーン判定失敗

**対応するPitfall**: Pitfall 4（トーンミスマッチ）

### 検出条件
- Q3 トーンキャリブレーションでトーンマトリクスとのミスマッチを検出
- 感情シグナル `angry` × セグメント `Complaint-in-progress` に対して共感文が含まれていない
- VIPセグメントへの標準テンプレートそのままの回答

### 影響
- Medium〜High — 顧客満足度低下、クレームの激化、チャーンリスク

### 対応手順

```
Step 1: tone-guidelines.md のトーンマトリクスで正しいトーンを確認
Step 2: 自動調整を試みる（フレーズ置換・丁寧度調整）
Step 3: 自動調整後、再度トーンマトリクスと照合
Step 4: 照合結果が「即エスカレーション」セルの場合 → Q3→R4 リペアパスを実行
        （注: リペアパスは最大1回。repair_count >= 1 の場合は強制エスカレーション）
Step 5: 調整後の回答を Q1 から再検証
Step 6: 監査ログにトーン調整内容を記録
```

### GOOD / BAD

**GOOD**: `angry` × `Complaint-in-progress` の顧客に対し、Q3でトーン不一致を検出。「即エスカレーション」セルと判定し、R4へリペアパスを実行した。

**BAD**: `angry` × `Complaint-in-progress` の顧客に「お手続きは以下の通りです」と事務的なメールを送信した。
→ 理由: Domain Research Pitfall 4（トーンミスマッチ）。共感・謝罪のない事務的回答はクレームを激化させる。

## シナリオ 6: Q3 リペアパスエラー

### 検出条件
- Q3→R4 リペアパス実行後、再度 Q3 で同じ問題（即エスカレーション判定）が発生
- `quality.q3_repair_count >= 1` かつ Q3 で再度エスカレーション判定

### 影響
- High — リペアループによる処理遅延、SLA超過リスク

### 対応手順

```
Step 1: repair_count を確認（>= 1 の場合は以下を実行）
Step 2: 自動調整を停止し、強制エスカレーション（R4）に移行
Step 3: 引き継ぎサマリーに「Q3リペアパス上限到達」を明記
Step 4: 顧客への暫定回答を送信
Step 5: 監査ログにリペアパスエラーを記録
Step 6: マネージャーへ通知（品質問題の可能性）
```

### GOOD / BAD

**GOOD**: Q3→R4 リペアパスを1回実行後、再度 Q3 で同じ問題が発生。repair_count = 1 のため自動調整を停止し、強制エスカレーションへ移行した。

**BAD**: repair_count を確認せず、Q3→R4 リペアパスを3回繰り返した。
→ 理由: ループ制御上限（最大1回）を超える処理は監査ログ上の問題になり、SLA超過にも直結する。

## エラー監査ログ形式

```json
{
  "ticket_id": "TKT-20240415-001",
  "error_scenario": "kb_miss | pii_leak | escalation_unavailable | sla_exceeded | tone_mismatch | repair_path_error",
  "detected_at": "{ISO 8601 timestamp}",
  "stage": "{Q1 | R2 | Q3 | ...}",
  "action_taken": "{実施した対応内容}",
  "escalated_to": "{エスカレーション先 or null}",
  "resolution": "resolved | pending | human_review_required"
}
```
