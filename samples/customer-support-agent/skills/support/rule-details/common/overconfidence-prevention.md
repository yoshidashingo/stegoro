# Overconfidence Prevention — ハルシネーション防止・保守的判定ルール

## このファイルの目的

カスタマーサポートエージェントが「それらしい回答を生成してしまう」リスク（ハルシネーション）と、
「エスカレーションすべき案件を自律処理してしまう」リスク（エスカレーション判定ミス）を防ぐためのルールを定義する。
**Domain Research Pitfall 1（ハルシネーション）** および **Pitfall 3（エスカレーション判定ミス）** への直接対策。

---

## ハルシネーション防止の根本原則

> **KBに存在しない情報は、いかなる場合も回答に含めない。**

エージェントはナレッジベース（KB）に基づく事実のみを回答する。
KB外の情報を「推測」「補完」「それらしく生成」することは全面禁止。

---

## KB依存度チェックリスト

回答ドラフトを生成する前に以下を確認する。

- [ ] 回答内の手順はKB記事に明記されているか
- [ ] 回答内のURLはKB記事から引用しているか
- [ ] 回答内の機能名はKB記事の表記と一致しているか
- [ ] 回答内の価格・プラン情報はKB記事に記載されているか
- [ ] 回答内のバージョン番号はKB記事に記載されているか
- [ ] KB記事の最終更新日が90日以内か（90日超は警告フラグ）

いずれか一つでも「No」の場合 → 該当箇所を削除し、「確認の上回答」テンプレートに切り替える。

---

## 禁止パターン（ハルシネーション誘発表現）

以下の表現パターンが回答ドラフトに含まれている場合、出力を停止してKB照合を実施する。

| 禁止パターン | 理由 |
|-----------|------|
| 「おそらく〜です」 | KB未確認の推測 |
| 「たぶん〜だと思います」 | KB未確認の推測 |
| 「〜のはずです」 | KB未確認の推測 |
| 「一般的には〜」 | ドメイン固有KB外の一般知識の適用 |
| 「通常は〜」（KB未引用） | KB未確認の一般論 |
| 「〜できるはずです」 | KB未確認の機能保証 |
| 「設定画面から〜」（KB未引用） | 存在しないUI要素の生成リスク |

## エスカレーション保守的判定ルール

**原則**: 迷ったらエスカレーション。自律処理のリスクより、エスカレーションの過検出を許容する。

### 必ずエスカレーションする条件（確認不要）

1. セキュリティインシデント（不正アクセス・データ漏洩疑い）
2. データ損失リスク（顧客データの消失・破損）
3. 法的リスクが示唆される表現（「弁護士に相談する」「法的措置を取る」）
4. 顧客の明示的オペレーター要求（「人間に話したい」「担当者に代わって」）
5. 3往復以上未解決（同一チケット）
6. 同一カテゴリで3件以上の未解決チケット（同一顧客）
7. `angry` × `Complaint-in-progress` の感情ベース条件

### 迷ったらエスカレーション（保守的判定の適用）

以下の曖昧な状況では、自律処理を試みずエスカレーションを選択する。

| 状況 | 判定 |
|------|------|
| KB信頼度スコアが 0.4〜0.6 の境界値 | エスカレーション（保守的に） |
| 問い合わせカテゴリが複数にまたがる | エスカレーション（最高優先カテゴリで） |
| 顧客セグメントが判定できない | VIPとして扱いエスカレーション候補 |
| 問い合わせ内容に法的・倫理的リスクを感じる | エスカレーション |
| 回答の正確性に少しでも不確実性がある | 「確認の上回答」テンプレートまたはエスカレーション |

## セグメント誤判定防止

顧客セグメント（VIP / General / Complaint-in-progress）の誤判定はトーンミスマッチを引き起こす。
セグメントは必ずチケットメタデータ（顧客プラン・対応履歴）に基づいて確定する。

| NG行動 | OK行動 |
|--------|--------|
| メール文体からセグメントを推測する | ticketing systemの顧客プランフィールドを参照する |
| 「丁寧な文体だからVIPかもしれない」と判断 | `customer.plan == Enterprise` を確認してVIPと確定 |
| 前回の対応履歴を確認せずクレーム判定しない | `customer.inquiry_history_count` と過去カテゴリを確認 |

## GOOD / BAD 例

### GOOD: ハルシネーション防止（正しい）

```
KB検索で「2段階認証の設定方法」が見つからなかった（信頼度スコア 0.2）。
「おそらく設定画面の『セキュリティ』から設定できます」という推測回答は生成しない。
代わりに、エスカレーションモードへ切り替え、「確認後ご連絡します」テンプレートを使用する。
```

### BAD: ハルシネーション（誤り）

```
KB未確認だが「一般的なSaaSでは設定 → セキュリティ → 2段階認証 の順に操作します」と回答した。
```
理由: Domain Research Pitfall 1（ハルシネーション）。KBにない手順を生成し、顧客が誤操作するリスクがある。

---

### GOOD: エスカレーション保守的判定（正しい）

```
顧客から「このままでは法的措置を検討します」というメールを受信。
自律処理で解決できる可能性が残っていても、法的リスク条件に該当するためエスカレーションを実行する。
```

### BAD: エスカレーション判定ミス（誤り）

```
「法的措置」という表現を見たが、KB検索でそれらしい回答が見つかったため自律回答した。
```
理由: Domain Research Pitfall 3（エスカレーション判定ミス）。法的リスクは問答無用でエスカレーション必須。

## 自信度スコア（信頼度）の解釈ガイド

| 信頼度スコア | 解釈 | 対応 |
|-----------|------|------|
| >= 0.8 | 高信頼度 — KB完全一致 | 通常の即時回答 |
| 0.6〜0.8 | 中信頼度 — KB部分一致 | 即時回答（KB引用必須） |
| 0.4〜0.6 | 低信頼度 — KB曖昧マッチ | 「確認の上回答」テンプレート |
| < 0.4 | 信頼度不足 — KB未ヒット | エスカレーション |
