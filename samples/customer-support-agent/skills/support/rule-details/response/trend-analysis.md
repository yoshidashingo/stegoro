# R5: Trend Analysis — Phase Rule

## Purpose

管理者リクエストまたは定期レポートタイミングで発動し、問い合わせ傾向を集計・分析してレポートを生成する。レポートドラフトは Q1〜Q4 の品質チェック（PII スキャンを含む）を通過してから出力する。CP-4 でレポートの承認を求める。

**Classification**: CONDITIONAL EXECUTE
**Execute IF**: 管理者が分析モードをリクエスト / 定期レポートタイミング（週次・月次バッチ）
**Skip IF**: 即時回答モード / エスカレーションモード
**Approval Gate**: あり（CP-4: 分析レポートのレビュー）
**参照元**: `core-workflow.md` R5 セクション

---

## Prerequisites

- R1 で分析モードが確定していること
- 集計対象期間のチケットデータ（クローズ済み・オープン含む）にアクセス可能であること
- CSAT スコアデータにアクセス可能であること（利用可能な場合）
- `common/output-structure-patterns.md` を読み込み済み（レポートテンプレートの参照元）

---

## Execution Steps

### Step 1: 集計対象期間と軸の確認

管理者リクエストまたはシステムトリガーから集計パラメータを取得する。

**集計対象期間**（デフォルト値）:

| レポート種別 | 対象期間 |
|-----------|---------|
| 週次レポート | 直近7日間（月曜日〜日曜日） |
| 月次レポート | 直近1ヶ月（1日〜末日） |
| 随時レポート | 管理者指定の期間 |

**集計軸**:

| 軸 | 説明 |
|---|-----|
| カテゴリ軸 | primary_category × subcategory |
| 優先度軸 | P1〜P4 の件数・比率 |
| セグメント軸 | VIP / General / Complaint-in-progress 別件数 |
| プラン軸 | Free / Pro / Enterprise 別件数 |
| 期間軸 | 日次・週次の推移グラフデータ |
| モード軸 | 即時回答 / エスカレーション / 分析の件数・比率 |

### Step 2: KPI ダッシュボード集計

以下の KPI を算出する。

| KPI | 計算方法 | 目標値（参考） |
|----|---------|------------|
| FRT（First Response Time）平均 | 受信〜最初の応答の平均時間（営業時間ベース） | SLA テーブルの各プラン FRT 以内 |
| FRT SLA 達成率 | SLA 内に応答したチケット数 / 全チケット数 × 100 | >= 95% |
| 解決率（Resolution Rate） | クローズ済みチケット数 / 全チケット数 × 100 | >= 85% |
| エスカレーション率 | エスカレーションチケット数 / 全チケット数 × 100 | <= 20% |
| CSAT 平均スコア | 回答済み CSAT スコアの平均値 | >= 4.0 / 5.0 |
| 反復問い合わせ率 | repeat_count >= 2 のチケット数 / 全チケット数 × 100 | <= 10% |

### Step 3: トレンド検出

以下のパターンを検出し、レポートに「注目トレンド」として記録する。

**急増カテゴリの検出**:
- 前期比 150% 以上の増加: 「急増」フラグ
- 前期比 200% 以上の増加: 「急増（要対応）」フラグ + 推奨アクション付与

**反復問い合わせパターンの検出**:
- 同カテゴリ・サブカテゴリで repeat_count >= 2 が週 5 件以上: 「KB ギャップ候補」として記録（F2 トリガー条件）

**CSAT 低スコアクラスターの検出**:
- CSAT <= 3 のチケットを カテゴリ × 担当モード でクラスタリング
- 同クラスター内で 3 件以上: 「対応品質課題」として記録

**SLA 超過傾向の検出**:
- SLA 超過率が前期比 120% 以上: 「SLA リスク」として記録

### Step 4: レポートドラフト生成

以下のテンプレートに従いレポートドラフトを生成する。

```
==== カスタマーサポート {週次 | 月次 | 随時} レポート ====
対象期間: {開始日} 〜 {終了日}
生成日時: {ISO 8601}

## 1. サマリー

- 総チケット数: {N} 件
- FRT SLA 達成率: {%}
- 解決率: {%}
- エスカレーション率: {%}
- CSAT 平均: {スコア}
- 主な注目トレンド: {上位2〜3件のトレンドを箇条書き}

## 2. カテゴリ別集計

| カテゴリ | 件数 | 前期比 | エスカレーション率 | CSAT 平均 |
|---------|-----|------|----------------|---------|
| Security | {N} | {%} | {%} | {スコア} |
| Bug | {N} | {%} | {%} | {スコア} |
| Billing | {N} | {%} | {%} | {スコア} |
| Feature | {N} | {%} | {%} | {スコア} |
| Account | {N} | {%} | {%} | {スコア} |

## 3. 優先度・セグメント別集計

| 優先度 | 件数 | SLA 達成率 |
|------|-----|---------|
| P1（Critical） | {N} | {%} |
| P2（High） | {N} | {%} |
| P3（Medium） | {N} | {%} |
| P4（Low） | {N} | {%} |

## 4. 注目トレンド

{検出されたトレンドを箇条書きで記載}
例:
- 🔺 Billing/payment_failure が前期比 180%（急増）
- 📊 Bug/api_error の CSAT 平均 2.8（低スコアクラスター検出）
- ⚠️ repeat_count >= 2 の Billing チケットが週 7 件（KB ギャップ候補）

## 5. 推奨アクション

{各トレンドに対する具体的な推奨アクションを記載}
例:
- Billing/payment_failure 急増: 決済プロバイダーとの連携状況を確認。KB 記事「支払い失敗時の対処法」を更新する
- Bug/api_error CSAT 低下: API エラーレスポンスの改善を開発チームへエスカレーション
- KB ギャップ候補: F2（FAQ Candidate Extraction）でクラスタリング分析を実施

==== レポート終了 ====
```

**PII 除去の徹底（v2）**: レポートドラフトには個別顧客情報（メールアドレス・氏名・電話番号等）を一切含めない。集計データのみを記載する。Q1（PII Scan）で残存 PII を検出・除去する。

### Step 5: チケットへの記録

```yaml
trend_analysis:
  report_type: "{weekly | monthly | adhoc}"
  target_period:
    start: "{ISO 8601}"
    end: "{ISO 8601}"
  total_tickets: "{整数}"
  kpi_summary:
    frt_sla_achievement: "{%}"
    resolution_rate: "{%}"
    escalation_rate: "{%}"
    csat_average: "{スコア}"
  detected_trends:
    - type: "{surge | kb_gap | csat_cluster | sla_risk}"
      category: "{カテゴリ}"
      detail: "{詳細}"
  report_draft_generated: "{true}"

audit_log:
  - timestamp: "{ISO 8601}"
    stage: "R5"
    action: "trend_report_generated"
    details: "type: {report_type}, tickets: {total}, trends: {検出数}"
```

## Question Generation（CP-4）

R5 完了後、レポートドラフトを提示しユーザー確認を求める（CP-4）。

**CP-4 提示テンプレート**:

```
### REVIEW REQUIRED
傾向分析レポートを生成しました。内容を確認してください。

**レポート情報**
- 種別: {週次 | 月次 | 随時}
- 対象期間: {開始日} 〜 {終了日}
- 総チケット数: {N} 件
- 検出トレンド数: {N} 件

**注目トレンド（抜粋）**
{上位3件のトレンドを箇条書き}

※ この後、レポートは Q1〜Q4 の品質チェックを通過します（PII スキャンを含む）。

### WHAT'S NEXT
**A) 修正依頼** — レポート内容を修正する
**B) Q1（PII Scan）へ続行** — 品質チェックへ進む
```

## Output Artifacts

R5 が生成するアーティファクト:

1. **傾向分析レポートドラフト**（Q1〜Q4 通過後に管理者へ送付）
2. **F2 トリガー候補情報**（KB ギャップ候補・CSAT 低スコアクラスターが検出された場合）
3. **監査ログエントリ** — `stage: R5, action: trend_report_generated`

## Error Handling

### エラー 1: 顧客メールアドレスがレポートに残存（Pitfall: PII 漏洩）

**症状**: レポートの「注目トレンド」セクションに具体例として顧客のメールアドレスが記載されている。  
**原因**: Step 4 のレポート生成時に個別チケット情報を例示として埋め込んでしまう。  
**対処**: レポートには集計数値・カテゴリ・比率のみを使用する。個別顧客情報は一切含めない。Q1（PII Scan）がレポートの PII を検出した場合は該当箇所を除去してから出力する。

**GOOD例**:
```
注目トレンド:
「Billing/payment_failure が前期比 180%（14件→25件）に増加。
 支払い失敗後の再試行フローに関する問い合わせが集中。」
→ 個別顧客情報なし → Q1 PII Scan: 検出なし → そのまま出力
```

**BAD例**:
```
注目トレンド:
「Billing/payment_failure 急増の代表例:
 user@example.com（Enterprise）から『カードが通らない』との報告が4件。」
→ Q1 PII Scan: メールアドレス検出 → user@example.com を [REDACTED] にマスキング
→ しかし固有顧客情報の記載自体が不適切。Step 4 の生成段階で排除すべき。
```

### エラー 2: サンプル数不足で誤ったトレンド報告（Pitfall: ハルシネーション回答）

**症状**: 集計期間内の特定カテゴリが 2 件しかないのに「急増トレンド」として報告。前期も 1 件だったため前期比 200% となってしまう。  
**原因**: トレンド検出で件数の最小閾値を設けていない。  
**対処**: 「急増」フラグの発動条件に「件数 >= 5 件」の最小サンプル数閾値を追加する。サンプル数不足の場合は「参考（サンプル数不足）」として別セクションに記載し、主要トレンドとしては扱わない。

**GOOD例**:
```
Security カテゴリ: 期間内 3 件（前期 1 件 → 前期比 300%）
→ サンプル数 3 < 5（最小閾値）
→ 「急増」フラグ付与せず
→ レポートの「参考（サンプル数不足）」セクションに記載
→ 管理者への誤ったアラート発報を防止
```

**BAD例**:
```
Security カテゴリ: 期間内 3 件（前期比 300%）
→ 最小閾値なし → 「急増（要対応）」フラグ付与
→ 管理者が緊急対応を指示
→ 実際には 3 件のみで統計的に有意でない
```

### エラー 3: レポートが Q1-Q4 を通過せずに出力される（Pitfall: PII 漏洩）

**症状**: 分析レポートが CP-4 承認後に直接管理者へ送付され、Q1 PII Scan が実行されない。  
**原因**: 「分析レポートは顧客に送るものではないので品質チェック不要」という誤った判断。  
**対処**: `core-workflow.md` の「All Modes Through QUALITY」原則に従い、分析レポートも必ず Q1〜Q4 を通過する。R5 → Q1 → Q2 → Q3 → Q4（管理者へ送付）のフローを厳守する。
