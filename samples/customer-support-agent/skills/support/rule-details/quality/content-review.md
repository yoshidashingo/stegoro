# Q2: Content Review（コンテンツレビュー）

## Purpose

PIIスキャン済み出力ドラフトに対して、ナレッジベース（KB）との照合・禁止表現チェック・事実確認を実施する。KB記事の鮮度も確認し、90日以上更新されていない記事への依存に警告を発する。SLA残時間が15分未満の場合は簡易チェックパスに切り替える。

**Classification**: ALWAYS EXECUTE

**参照元**: `core-workflow.md` Q2、`common/content-validation.md`、`common/overconfidence-prevention.md`

---

## Prerequisites

- Q1（PII Scan）が完了し、マスキング済み出力ドラフトが渡されていること
- R2（Knowledge Retrieval）で取得したKB記事リスト（記事ID・最終更新日・信頼度スコア）が参照可能であること
- `common/content-validation.md` の禁止表現リストが読み込まれていること
- SLA残時間が確認可能であること

---

## Execution Steps

### Step 1: チェックモードの決定

```
SLA残時間チェック:
  残時間 >= 15分 → フルチェックモード（Step 2〜4をすべて実行）
  残時間 < 15分  → 簡易チェックモード（Step 2のみ。Step 3・4はスキップ）
  SLA超過        → 簡易チェックモード（Step 2のみ）
```

簡易チェックモードでログに `SLA_SHORTCUT: SIMPLIFIED_CHECK` を記録する。

### Step 2: KB照合チェック（フル・簡易共通）

R2で参照したKB記事IDに対し、ドラフト内の記述との整合を検証する。

**照合対象**:
- 手順の説明（ステップ番号・操作名）
- 機能名・サービス名
- URLリンク（KB記事URL、製品ページURL）
- 料金・プラン情報
- バージョン番号

**照合ロジック**:
1. ドラフト内の各記述をKB記事テキストと突き合わせる
2. KB記事に存在しない情報を含む記述を検出したら `UNVERIFIED_CLAIM` フラグを立てる
3. KB引用形式の確認: `([KB記事タイトル]: {URL})` 形式でない引用は `CITATION_FORMAT_ERROR` フラグを立てる

**ハルシネーション防止**（Pitfall 1対応）:
- `UNVERIFIED_CLAIM` フラグが1件でも存在する場合、当該記述を自動削除し `[要確認: KB未掲載情報が削除されました]` に置き換える
- KB外情報の生成を発見した場合、`common/overconfidence-prevention.md` のRed Flag対応手順を実行

### Step 3: 禁止表現チェック（フルチェックのみ）

`common/content-validation.md` の禁止表現リストに基づき全文走査する。

**禁止表現カテゴリ**:

| カテゴリ | 禁止例 | 置き換え案 |
|---------|--------|-----------|
| 断定的拒否 | 「できません」「不可能です」 | 「現在の仕様ではご対応が難しい状況ですが、代替として…」 |
| 責任転嫁 | 「そちらのミスです」「お客様の設定が原因です」 | 「設定を一緒に確認させていただけますか」 |
| 過剰約束 | 「必ず解決します」「100%大丈夫です」 | 「全力で対応いたします」 |
| 社内用語の露出 | 「チケット番号で検索して」「Slackで確認します」 | 削除または「社内で確認の上」に置き換え |
| SLA数値の誤引用 | 「24時間以内に必ず」（FreeプランのSLA数値を誤引用） | プラン別SLAを正確に引用 |

**自動処理**:
- 置き換え可能な表現 → 自動修正して `AUTO_FIXED` ログ
- 文脈依存で判断が必要な表現 → `REVIEW_REQUIRED` フラグ + 該当箇所を明記

### Step 4: KB記事鮮度再検証（フルチェックのみ）

R2で参照したKB記事それぞれについて最終更新日を確認する。

```
最終更新日チェック:
  更新日が現在から90日以内 → OK
  更新日が現在から90日超  → WARNING フラグ: KB_STALE_WARNING
  更新日が現在から180日超 → CRITICAL フラグ: KB_STALE_CRITICAL
```

**KB_STALE_WARNING 時の処理**:
1. 当該KB記事の記述を含む回答箇所に免責文を追加: `（※この情報は最新でない可能性があります。詳細は担当スタッフにご確認ください）`
2. KB管理チームへの更新依頼ジョブをログに記録

**KB_STALE_CRITICAL 時の処理**:
1. 当該KB記事への依存箇所を回答から削除
2. R2に戻らず（SLA優先）、「担当者が最新情報を確認の上ご回答します」テンプレートに差し替え
3. R4（エスカレーション）への切り替えをQ3に申告

### Step 5: チェック結果サマリー作成

```
[CONTENT REVIEW LOG]
- Timestamp: {ISO 8601}
- Ticket ID: {TKT-XXXXXXXX-NNN}
- Check Mode: FULL | SIMPLIFIED (SLA_SHORTCUT)
- KB Cross-check:
  - Articles Checked: {N}件
  - UNVERIFIED_CLAIM: {N}件 → 削除済み
  - CITATION_FORMAT_ERROR: {N}件 → 修正済み
- Prohibited Expression:
  - AUTO_FIXED: {N}件
  - REVIEW_REQUIRED: {N}件
- KB Freshness:
  - OK: {N}件
  - KB_STALE_WARNING: {N}件（90日超）
  - KB_STALE_CRITICAL: {N}件（180日超）→ エスカレーション申告
- Result: PASS | PASS_WITH_WARNINGS | ESCALATE_REQUIRED
```

## Question Generation

Q2は原則として自動処理であり、ユーザーへの確認は不要。ただし `REVIEW_REQUIRED` フラグの禁止表現については以下の確認を生成する。

```
### コンテンツレビュー確認
以下の表現が禁止表現に該当する可能性があります。

- 対象箇所: {文脈を示す前後15文字}
- フラグ理由: {禁止カテゴリ}
- 提案置き換え: {置き換え案}

A) 提案通りに修正して続行
B) 別の表現を指定して修正
```

## Error Handling

### エラー1: KB参照なしで回答が生成されている

**シナリオ**: R3またはR4のドラフトにKB引用が一切含まれず、すべての記述が `UNVERIFIED_CLAIM`

**原因候補**:
- Pitfall 1（ハルシネーション回答）: LLMがKB参照なしに「それらしい」内容を生成
- R2でKB信頼度スコア < 0.4 の状態が適切にエスカレーションへ渡されなかった

**対応**:
1. ドラフト全文を `[KBに基づく回答なし — 担当者が確認の上ご回答します]` に差し替え
2. R4（エスカレーション）への切り替えを申告
3. `HALLUCINATION_SUSPECTED` ログを記録

### エラー2: 廃止済み機能への言及検出

**シナリオ**: KB記事に「2024年3月31日をもって廃止」と記載されている機能の手順を回答に含めている

**対応**:
1. 廃止情報をKBから取得し、当該箇所を置き換え: `（この機能は{廃止日}に終了しました。代替として{代替機能}をご利用ください）`
2. 代替機能のKB記事URLを引用して付記
3. Pitfall 1（ハルシネーション）派生ケースとして記録

### エラー3: SLA簡易チェック中に重大な誤りを発見した場合

**シナリオ**: SLA残5分で簡易チェック中、KB照合で価格情報の明確な誤りを発見

**対応**:
1. 簡易チェックモードでも `UNVERIFIED_CLAIM` フラグは有効（価格情報の誤りは重大リスク）
2. 当該箇所を削除し「詳細な料金については担当者が改めてご案内します」に置き換え
3. SLA超過を受け入れ、正確な情報を優先

### エラー4: 分析レポート（R5出力）での顧客情報集計ミス

**シナリオ**: 傾向分析レポートに特定顧客の問い合わせ内容が個別に記載されており、集計データとして不適切

**原因候補**:
- Pitfall 2（PII漏洩）の変形: Q1でPIIがマスクされていても、内容から個人が特定可能な場合
- Pitfall 4（トーンミスマッチ）の関連: 顧客属性情報が不適切に集計

**対応**:
1. 個人特定可能な記述を集計データ（件数・傾向）に変換
2. 「特定顧客情報はレポートに含めない」ルールをQ3通過後も維持

## GOOD / BAD 具体例

### GOOD 例1: 廃止機能の検出と置き換え

**BAD（処理前）**:
```
旧ダッシュボードの「レポート出力」ボタンから月次レポートをご取得いただけます。
（KB記事「dashboard-report-export」参照）
```
KB記事の最終更新日: 2023年11月20日（現在から185日超 → KB_STALE_CRITICAL）
KB記事本文: 「この機能は2024年2月1日に廃止されました」

**GOOD（Q2処理後）**:
```
レポートの取得方法が変更されました。新しい「アナリティクス」メニューからご取得いただけます。
（[アナリティクス機能のご案内]: https://kb.example.com/analytics-guide）
（※この情報は最新でない可能性があります。詳細は担当スタッフにご確認ください）
```

### GOOD 例2: KB_STALE_WARNING での免責文追加

**シナリオ**: KB記事「price-list-2023q4」最終更新2023年10月（95日前）を参照した回答

**BAD（警告なし）**:
```
Proプランは月額 9,800円です。
```

**GOOD（警告付き）**:
```
Proプランは月額 9,800円です。
（[料金プランページ]: https://example.com/pricing）
（※この情報は最新でない可能性があります。詳細は担当スタッフにご確認ください）
```

### BAD 例1: SLA圧力でKB照合をスキップ

**BAD（NG行動）**:
SLA残8分のため「時間がないのでKB照合はスキップ」として、ハルシネーション記述を含むドラフトをそのままQ3へ渡す。

**結果**: 存在しない機能「自動バックアップ（月額+500円）」を案内。顧客から後日クレーム。Pitfall 1（ハルシネーション）直撃。

**GOOD（正しい行動）**:
SLA残8分でも簡易チェックモード（Step 2のKB照合）は必ず実施。`UNVERIFIED_CLAIM` を削除して続行。

### BAD 例2: 禁止表現を見逃し

**BAD（見逃し）**:
```
ご申告の内容ですが、こちらはお客様のパスワード設定が原因と考えられます。
設定をご確認いただけますか。
```
「お客様の〜が原因」は責任転嫁の禁止表現。

**GOOD（修正後）**:
```
ご不便をおかけして申し訳ございません。パスワード設定を一緒に確認させていただけますでしょうか。
```
責任転嫁表現を削除し、共同解決のトーンに変換。
