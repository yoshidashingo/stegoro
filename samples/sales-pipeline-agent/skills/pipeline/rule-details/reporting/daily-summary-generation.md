# R1: Daily Summary Generation — Phase Rule

## Purpose

当日の対応状況・翌日優先タスクリスト・パイプラインヘルスサマリーを生成し、通知階層化ルールに基づいて配信する。緊急通知はSlack DM即時送信、重要・情報はサマリー集約とする。同一商談への重複通知を4時間以内で抑制し（Pitfall 3対策）、CRMデータ鮮度警告でSalesforceデータ品質を監視する（Pitfall 1対策）。

**Classification**: ALWAYS EXECUTE（全サイクルで必須）
**Approval Gate**: あり（CP-6: 通知階層化・重複抑制が適用されているか確認）
**参照元**: `workflow-architecture.md` R1 セクション

---

## Prerequisites

- D5のロールバック検証結果が確定していること（または S4で検出なしの場合）
- S3の統合ビュー・S4の検出結果・A1のヘルススコアが参照可能であること
- D1・D3の実行完了記録が参照可能であること
- 前回通知ログ（`state/last-notification-log.json`）が参照可能であること（重複抑制用）
- 配信先設定（`config/notification-settings.yaml`）が設定済みであること

---

## Execution Classification

**ALWAYS EXECUTE**

- Execute IF: 常に実行（DISPATCHサイクル完了後、またはS4で検出なし）
- Skip IF: なし（スキップ不可）

## Execution Steps

### Step 1: 通知の優先度分類と重複抑制チェック

当日検出・対応した全案件を3段階の通知優先度に分類する。

**通知優先度階層**（Pitfall 3対策）:

| 優先度 | 条件 | 配信方法 |
|--------|------|---------|
| 緊急 | クロージング日24時間以内の異変、¥10M以上商談の停滞検出 | 担当者Slack DM即時送信 |
| 重要 | 停滞商談検出、未返信フォロー必要、期限7日以内 | サマリーに集約（次回通知） |
| 情報 | Activity自動記録完了、ステータス更新完了、会議準備生成完了 | 日次ダイジェストに集約 |

**重複通知抑制チェック**（Pitfall 3対策）:
```
FOR each notification IN notification_candidates:
  last_sent = last_notification_log.get(notification.opportunity_id)

  IF last_sent IS NOT NULL:
    elapsed = current_time - last_sent.sent_at

    IF notification.priority == "緊急":
      # 緊急は重複抑制なし（常に送信）
      INCLUDE

    ELSE IF elapsed < 4時間:
      EXCLUDE  # 重複通知として除外
      log("重複通知抑制: {opportunity_name} (前回 {elapsed}分前に送信)")

    ELSE:
      INCLUDE
```

### Step 2: 緊急通知の即時送信

緊急優先度の案件を担当者Slack DM に即時送信する。

**緊急通知フォーマット**:
```
🚨【緊急】パイプラインアラート

対象商談: {opportunity_name}
金額: ¥{amount}M / ステージ: {stage} / クロージング: {close_date}（{days_to_close}日後）
ヘルススコア: 赤（{health_score}点）

検出内容: {検出種別の説明}
推奨アクション: {action_type の説明}

Salesforceで確認: {sf_opportunity_url}
```

緊急通知を送信した案件は `last_notification_log` に記録する。

### Step 3: 日次サマリーの構築

当日の全対応状況を集約したサマリーを構築する。

**サマリー構造**:
```markdown
## 営業パイプライン対応サマリー {YYYY/MM/DD}

### 対応完了
- 自動実行: {X}件
  - Activity記録: {X}件
  - 担当者リマインダー送信: {X}件
  - 会議準備サマリー生成: {X}件
- 承認済み実行: {X}件
  - フォローアップメール送信: {X}件
  - ステージ更新: {X}件
  - リード登録: {X}件

### 承認待ち（次サイクル持ち越し）
{承認待ちアクションリスト（件数 <= 5件は全件表示、5件超は上位5件 + 残X件）}
- 例: 株式会社A — フォローアップメール送信 — 送信時刻: 10:10

### 次サイクル繰り延べ
- ループ上限・タイムアウト: {X}件
- 承認タイムアウト（2時間）: {X}件

### 翌日の優先タスク（上位5件）
{A3の優先度スコア降順 + P1アクションを必ず含む}
1. [P1] 商談A（¥8M、クロージング明日）— クロージング最終確認
2. [P2] 商談B（¥3M、Proposal14日停滞）— フォローアップメール承認待ち
3. [P2] 商談C（¥5M、期限5日）— ステージ更新提案
4. [P3] 商談D（¥2M、未返信3日）— フォローアップリマインダー
5. [P3] 商談E（¥4M、未追跡リード）— リード登録提案

### パイプラインヘルスサマリー
- 🔴 赤（要緊急対応）: {X}件 / 🟡 黄（注意）: {X}件 / 🟢 緑（良好）: {X}件
- 加重パイプライン合計: ¥{X}M
- トレンド: {改善 / 横ばい / 悪化}（前サイクル比）

{CRMデータ鮮度警告セクション（条件付き）}
```

**CRMデータ鮮度警告（Pitfall 1対策）**:
```
IF count(opportunities WHERE LastModifiedDate < today - 30日) >= 5:

### ⚠️ CRMデータ更新推奨
Salesforceで30日以上更新されていない商談が{X}件あります。
データを更新することで停滞検出の精度が向上します。
対象商談: {上位3件のリスト}
```

**障害ソースの透明性記録（Pitfall 4対策）**:
```
IF S2で障害ソースがあった場合:

### ℹ️ 今サイクルの制約
一部データソースで障害が発生しました。以下の検出カテゴリはスキップされました:
- Gmail障害 → 未返信メール検出をスキップ
次サイクルで再試行します。
```

### Step 4: サマリーの配信

構築したサマリーを設定された配信先に送信する。

**配信先の設定**（`config/notification-settings.yaml`）:
- チームサマリー: `#sales-pipeline` Slackチャンネル（全担当者向け集約ビュー）
- 個人タスクリスト: 各担当者Slack DM（自分担当の翌日タスクのみ抽出）

**個人タスクリストの抽出ロジック**:
- `action.owner_id == user.slack_id` でフィルタリング
- 担当者ごとに翌日タスクリストをパーソナライズ

**配信失敗時のフォールバック**:
- Slack配信失敗 → 担当者メールアドレスへサマリーをテキスト形式で送信
- 両方失敗 → 監査ログに記録し、次サイクルで再送を試みる

### Step 5: 通知ログの更新

送信した全通知を `state/last-notification-log.json` に記録する。

```json
{
  "notifications": [
    {
      "opportunity_id": "006xxxxxxxxxxxx",
      "notification_type": "urgent_alert",
      "sent_at": "2026-04-05T10:50:00+09:00",
      "channel": "slack_dm",
      "recipient": "U123456789"
    },
    {
      "opportunity_id": "006yyyyyyyyyyyyy",
      "notification_type": "summary_included",
      "sent_at": "2026-04-05T10:50:00+09:00",
      "channel": "slack_channel",
      "recipient": "#sales-pipeline"
    }
  ]
}
```

## Error Handling

### エラー 1: 重複通知による担当者の通知疲れ（Pitfall 3対策）

**症状**: 同一商談に対する「停滞検出」通知が、2時間前の「未返信検出」通知と重複して個別に送信される。担当者が重要通知を見落とす。
**原因**: Step 1の重複通知抑制チェックが実装されていない。
**対処**: `last_notification_log` に記録された前回通知から4時間以内の通知を抑制する。緊急通知のみ例外とする。

**GOOD例**:
```
商談F: 「未返信通知」を 08:30 に送信済み
→ 10:30 に同一商談の「停滞通知」を生成
→ 重複抑制チェック: elapsed = 2時間 < 4時間 → 抑制
→ サマリーには「停滞（重複抑制）」として集約のみ
→ 担当者の受信通知: 1件（重複なし）（Pitfall 3対策）
```

**BAD例**:
```
商談F: 「未返信通知」を 08:30 に送信
→ 10:30 に「停滞通知」も個別送信（重複抑制なし）
→ 担当者が2時間で同一商談の通知を2件受信
→ 重要通知が大量通知に埋もれて見落とし（Pitfall 3対策漏れ）
```

### エラー 2: CRMデータ鮮度劣化の検出漏れ（Pitfall 1対策）

**症状**: Salesforceに30日以上更新されていない商談が10件あるが、サマリーに警告が含まれず、営業担当がデータ更新を怠り続ける。停滞検出の精度が低下する。
**原因**: Step 3のCRMデータ鮮度警告条件チェックが実装されていない。
**対処**: 毎サイクルの集計時に30日以上未更新の商談件数を算出し、5件以上の場合は必ずサマリーに警告セクションを追加する。

**GOOD例**:
```
Salesforce確認: 30日以上未更新の商談 = 8件
→ 閾値（5件）超過 → サマリーに「CRMデータ更新推奨」セクションを追加
→ 対象商談上位3件をリストアップ
→ 担当者がデータ更新を実施 → 次サイクルの停滞検出精度が向上（Pitfall 1対策）
```

**BAD例**:
```
30日以上未更新の商談 = 8件（CRMデータ鮮度警告なし）
→ 停滞検出: Salesforceのみで判定 → 8件全てが停滞候補に
→ 毎サイクルに大量の停滞アラートが発生
→ 担当者が通知疲れで全アラートを無視（Pitfall 1・3対策漏れ）
```

### エラー 3: 障害ソースのスキップを通知しない（Pitfall 4対策）

**症状**: Gmail障害により未返信メール検出をスキップしたが、サマリーにその旨が記載されない。担当者が「未返信フォローが今日は検出されなかった」と誤解し、手動フォローアップを怠る。
**原因**: Step 3の障害ソース透明性記録が実装されていない。
**対処**: S2で障害ソースがあった場合は、サマリーの末尾に「スキップした検出カテゴリ」を必ず記載する。

**GOOD例**:
```
Gmail障害（S2でauth_error）→ 未返信メール検出をスキップ
→ R1サマリーに「ℹ️ Gmail障害のため未返信メール検出をスキップしました」を記載
→ 担当者が手動でGmailを確認 → 未返信フォローを自主実施
→ 商談機会の損失を防止（Pitfall 4対策）
```

**BAD例**:
```
Gmail障害 → 未返信メール検出スキップ（サマリー未記載）
→ 担当者: 「今日は未返信メールがなかった」と誤解
→ 手動フォローアップなし
→ 3件の重要なフォローアップが翌日まで放置
```

### エラー 4: 翌日優先タスクリストのP1欠落

**症状**: 翌日の優先タスクリストをランダムに上位5件を選んだため、P1アクション（クロージング翌日・¥15M大型商談）がリストに含まれなかった。担当者が最重要タスクを見落とした。
**原因**: Step 3のP1必須包含ロジックが実装されていない。
**対処**: 翌日優先タスクリストは必ず全P1アクションを含め、その後PriorityScore降順で上位5件になるよう構成する。P1が5件を超える場合は全P1を表示（5件制限を超えて表示）。

**GOOD例**:
```
翌日タスク候補: P1×2件 + P2×8件
→ P1（2件）を必須包含
→ 残り3枠にP2上位3件を追加
→ 翌日タスクリスト: P1×2 + P2×3（P1欠落なし）
```

**BAD例**:
```
翌日タスク候補: P1×2件 + P2×8件
→ PriorityScore降順ソートのみ（P1優先保証なし）
→ P1のスコアが P2より低かった場合（期限は明日だがAmount小）
→ P1がリストに入らない
→ 担当者が最重要タスクを見落として翌朝に混乱
```
