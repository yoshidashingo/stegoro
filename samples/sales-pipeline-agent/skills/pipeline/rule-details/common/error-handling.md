# Error Handling — 5 Pitfalls対応とエラー処理

## 基本方針

エラーは「無視」も「即時停止」もしない。
影響範囲を最小化しながら処理を継続し、担当者・管理者に必要な情報を通知する。
Salesforceをマスターデータソースとして扱い、商談データへの不可逆な変更には
特に慎重な対処を行う。

---

## Pitfall 1: CRMデータの鮮度劣化

### 症状
- Salesforceのレコードが実際の商談状況と乖離している
- 最終更新日から30日以上経過した商談が存在する
- アクティビティログとSalesforceステージの不整合が発生している

### 検出ロジック
```
検出条件:
  - Opportunity.LastModifiedDate < 現在日時 - 30日
  - かつ StageName NOT IN ('Closed Won', 'Closed Lost')
  → データ鮮度劣化フラグを付与

不整合チェック:
  - Gmail/Calendar/Slackで直近7日以内にアクティビティあり
  - かつ SalesforceのLastActivityDate >= 7日前でない
  → 「Salesforce未更新の可能性」フラグを付与
```

### 対処法
```
ステップ1: 更新リマインダーを自動送信（低リスク・自動実行）
  → 担当者Slack DM:「{商談名}のSalesforceが{N}日間未更新です。現状を更新してください」

ステップ2: 停滞判定時は多層判定を必須化
  → Salesforceのステージ滞留日数のみでの停滞判定を禁止
  → Gmail/Calendar/Slackのアクティビティを必ず確認する

ステップ3: 3サイクル連続で未更新の場合
  → マネージャーへのエスカレーション通知
  → 日次サマリーに「データ品質アラート商談リスト」を追記
```

## Pitfall 2: 過度な自動化による信頼失墜

### 症状
- 担当者の意図に反するSalesforceステータス更新が発生した
- 不適切なタイミングでの顧客向けメールが自動送信された
- 承認却下率が30%を超えている（自動判定ロジックの精度低下サイン）

### 検出ロジック
```
事前防止チェック（DISPATCHフェーズで実施）:
  - アクション種別が高リスクリストに含まれる → 必ず承認制を経由
  - 承認なしで高リスクアクションを実行しようとしている → 即時中断

事後検出:
  - 承認却下率 = 却下件数 / (承認件数 + 却下件数) >= 0.3
  → 自動判定ロジックの見直しフラグを生成
  → 管理者に「承認却下率が{N}%に達しました。判定ロジックの確認を推奨します」と通知
```

### 対処法
```
誤自動実行検出時（ロールバック処理）:
  ステップ1: rollback_snapshotsから実行前の状態を取得
  ステップ2: Salesforce APIでロールバック実行
    → ロールバック成功: 担当者に「{アクション}を取り消しました。元の状態に戻しました」と通知
    → ロールバック失敗: 管理者緊急通知 + 手動修正ガイドを提示 + 該当アクションの自動実行を停止

予防措置:
  - 高リスクアクションリストの定期レビューを促す（月次）
  - 承認却下理由のログ分析で誤判定パターンを特定
```

## Pitfall 3: 通知疲れ（Alert Fatigue）

### 症状
- 承認リクエストへの平均応答時間が1時間を超えている
- 同一商談への重複通知が発生している
- 担当者から「通知が多すぎる」というフィードバックがある

### 検出ロジック
```
重複通知チェック（送付前に確認）:
  - pending_approvalsに同一商談の未処理承認リクエストが存在する
  → 追加送付を禁止（既存リクエストへの返信を促すメッセージのみ送付）

応答率モニタリング:
  - 承認リクエスト送付後2時間以内の応答率 < 50%（週次チェック）
  → 「日次ダイジェスト移行」オプションを担当者に提案
```

### 対処法
```
通知の優先度階層化:
  緊急（即時通知）: クロージング期限7日以内・大型商談（¥10M以上）の異変
  重要（次回巡回サマリー）: 停滞検出・未返信フォロー・データ品質アラート
  情報（日次ダイジェスト）: Activity自動記録完了・ステータス更新完了

重複抑制ルール:
  - 同一商談への承認リクエストは未処理分が存在する間は追加送付しない
  - 同一カテゴリのアラートは24時間以内に同一担当者へ3件以上送付しない
  - 日次ダイジェストに集約可能な通知は個別送付を避ける
```

## Pitfall 4: API制限への対応不足

### 症状
- Salesforce APIのガバナーリミットに到達して巡回が中断した
- `REQUEST_LIMIT_EXCEEDED` エラーが発生した
- 日次API呼び出し上限の80%を超えた

### 検出ロジック
```
API使用量モニタリング（毎サイクル開始時）:
  GET /services/data/v59.0/limits からDailyApiRequestsを取得

  使用率 < 80%:  normal mode（通常巡回）
  使用率 >= 80%: safe_mode（巡回頻度を2時間に延長、バルクAPI優先）
  使用率 >= 95%: readonly mode（DML操作停止、読み取り専用で続行）
  使用率 = 100%: 巡回停止・管理者緊急通知
```

### 対処法
```
セーフモード時の動作変更:
  - 巡回間隔: 1時間 → 2時間
  - 大量レコード取得: バルクAPI 2.0を使用
  - SELECT文: 必要最小限のフィールドのみ
  - DML操作: 1回のリクエストにバッチ処理でまとめる（最大150件/トランザクション）

読み取り専用モード時の動作変更:
  - DML操作（INSERT/UPDATE/DELETE）を完全停止
  - 読み取りと検出のみ継続
  - 実行予定だったDML操作を繰り延べリストに追加

リトライロジック（指数バックオフ）:
  初回リトライ: 1秒後
  2回目: 2秒後
  3回目: 4秒後
  4回目: 8秒後
  5回目: 16秒後（最大）
  5回失敗: エラーログ記録・操作を繰り延べリストに追加
```

## Pitfall 5: データソース間の不整合

### 症状
- Salesforceでは停滞中だが、Slackでは活発にやり取りしている商談がある
- Gmail返信済みなのにSalesforceのアクティビティが未記録
- ソース間でクロージング日や金額が異なる

### 検出ロジック
```
矛盾検出パターン:
  パターンA: SF停滞 × Slack/Gmail活発
    → Salesforce.LastActivityDate >= 14日前
    → かつ Gmail/Slack.last_activity < 7日前
    → 判定: 「SF未更新の可能性」フラグ付与。停滞判定を保留

  パターンB: ソース間のクロージング日不整合
    → Salesforce.CloseDate と Gmail本文/Slackメッセージ内の日付が異なる
    → 判定: 「クロージング日不整合」フラグ付与。担当者に確認依頼

  パターンC: 金額不整合
    → Gmail/Slack内に言及された金額がSalesforce.Amountと20%以上乖離
    → 判定: 「金額不整合」フラグ付与。金額変更承認リクエストを生成
```

### 対処法
```
矛盾フラグ付与ルール:
  → 自動判定を保留し「要人間確認」フラグを付与
  → ASSESSMENTフェーズで承認リクエストに矛盾内容を明記
  → 「Salesforceでは{状態A}ですが、{ソース}では{状態B}が確認されました」と説明

情報優先順位（矛盾解消の基準）:
  1位: Salesforce（マスターデータソース）
  2位: Gmail（タイムスタンプが最新のもの）
  3位: Google Calendar
  4位: Slack（補助情報として参照）

ただし、Salesforceが30日以上未更新の場合は他ソースを優先して矛盾を報告する
```

## その他のエラーパターン

### OAuth トークン失効

```
検出: HTTP 401 INVALID_SESSION_ID

対処:
  ステップ1: リフレッシュトークンでアクセストークンを更新
  ステップ2: リフレッシュ成功 → 操作を1回リトライ
  ステップ3: リフレッシュ失敗（リフレッシュトークン失効）
    → 当該ソースを障害扱い
    → フォールバック処理へ移行
    → 管理者にアラート:「{ソース名}のOAuth認証が失効しました。再認証が必要です」
```

### ロールバック失敗

```
検出: ロールバック実行後にSalesforceのレコードが元に戻っていない

対処:
  ステップ1: 管理者に緊急通知
    「{商談名}のSalesforceデータを元に戻せませんでした。手動確認が必要です。
     変更前の状態: {before_state}」
  ステップ2: 手動修正ガイドを提示
  ステップ3: 該当アクション種別の自動実行を一時停止
    → 管理者が確認・解除するまで承認制に格上げ
```

## GOOD/BAD 例

**GOOD**: Salesforce APIの使用率が82%に達したため、自動的にセーフモードに切り替えた。
巡回間隔を2時間に延長し、以降のSOQLクエリでSELECT *を廃止して必要フィールドのみ取得した。
当日のガバナーリミット超過を回避し、夜間の日次サマリー生成まで正常に動作した（Pitfall 4対策）。

**BAD**: 承認リクエストに返答がないまま同一商談への新たな停滞アラートが発生し、
2件目の承認リクエストを追加送付した。担当者は重複通知に混乱し、
最初の承認リクエストを見逃して期限切れになった。既存の未処理承認リクエストがある場合は
追加送付を禁止すること（Pitfall 3対策）。
