# Action Authority — 自動実行/承認制の境界定義

## 基本原則

このファイルは Pitfall 2（過度な自動化による信頼失墜）対策の中核ルールです。
エージェントが実行するすべてのアクションは「低リスク（自動実行可）」か
「高リスク（承認制）」のいずれかに分類される。

**判断に迷った場合は常に「承認制」を選択する。**
自動実行は「明示的に低リスクと定義されたアクションのみ」に限定する。

---

## 低リスクアクション（自動実行可）

以下のアクションは担当者の承認なしで自動実行する。

| # | アクション | 対象 | 実行条件 |
|---|-----------|------|---------|
| 1 | Salesforce Activityログ記録 | Gmail送受信・カレンダーミーティング・Slack言及 | 記録漏れを検出した場合。同一内容が24時間以内に未記録であること |
| 2 | 内部リマインダー送信（Slack DM） | 担当営業担当者 | 承認リクエスト30分タイムアウト時、ミーティング24時間前 |
| 3 | ステージの前進更新 | Salesforce商談 | 後退でない場合のみ（Prospecting→Qualification等）。金額変更を伴わないこと |
| 4 | 会議準備サマリー生成 | Salesforceレコード + Gmail + 過去議事録 | ミーティング24時間前。読み取り専用操作のみ。外部送信なし |
| 5 | パイプラインヘルスレポート生成 | 読み取り専用 | 日次サマリー・週次分析。Salesforceへの書き込みなし |
| 6 | データ品質劣化の更新リマインダー送信 | 担当者Slack DM | 必須フィールド未入力・30日以上未更新を検出した場合 |
| 7 | 重複リード/商談の検出通知 | 担当者Slack DM | 重複候補の提示のみ。マージ操作は禁止 |
| 8 | 日次・週次レポート生成・配信 | チームSlackチャンネル | 読み取り専用。Salesforceへの書き込みなし |

### 低リスクアクション実行前の必須チェック

自動実行前に以下を確認する。

```
□ 同一商談への自動実行が過去24時間以内に実施されていないか（過剰実行防止）
□ 変更対象フィールドが「低リスクフィールドリスト」に含まれるか
   低リスクフィールド: ActivityDate, Subject, Description, WhatId, WhoId
□ ステージ前進更新の場合: 後退方向でないことを確認
□ Activityログ記録の場合: 同一件名・同一日付のActivityが既に存在しないこと
□ rollback_snapshotを記録してから実行する
```

## 高リスクアクション（承認制）

以下のアクションは**必ず担当者の承認を得てから実行する**。
承認なしでの実行は技術的に防止する。

| # | アクション | 対象 | リスク根拠 |
|---|-----------|------|-----------|
| 1 | 顧客へのメール送信 | Gmail経由の外部送信 | 顧客との関係に直接影響。誤送信は信頼毀損（Pitfall 2） |
| 2 | 商談金額の変更 | Salesforce Opportunity.Amount | フォーキャスト・売上報告・コミッション計算への影響 |
| 3 | ステージの後退 | Salesforce Opportunity.StageName（前ステージへの変更） | 担当者モチベーション・パイプライン予測への影響 |
| 4 | 商談クローズ（Won/Lost） | Salesforce Opportunity（Closed Won/Closed Lost） | 売上計上・KPI・コミッションへの直接影響 |
| 5 | 新規リードの商談化 | SalesforceリードからOpportunityへの変換 | 誤変換は営業プロセスの歪み・重複商談の発生 |
| 6 | 既存レコードの削除 | SalesforceオブジェクトのDELETE操作 | 復元不可のデータ損失リスク |
| 7 | 重複レコードのマージ | Salesforceリード/商談のマージ | データ損失・履歴喪失のリスク |
| 8 | Salesforceカスタムフィールドの変更 | 標準外フィールド | 業務プロセスへの予期しない影響 |
| 9 | クロージング日の変更 | Salesforce Opportunity.CloseDate | フォーキャスト精度・優先度への影響 |
| 10 | 担当者の変更 | Salesforce Opportunity.OwnerId | 引き継ぎ・コミッション・通知設定への影響 |

### 承認リクエスト送付フロー

```
高リスクアクション検出
        ↓
承認リクエスト生成（question-format-guide.md のテンプレートを使用）
        ↓
担当者 Slack DM に送付
        ↓
pending_approvals に追加（timeout_at = 送付時刻 + 2時間）
        ↓
30分後: 未承認 → 自動リマインダー送付（reminder_sent = true）
        ↓
2時間後: 未承認 → deferred_actions に移動（defer_reason: approval_timeout）
        ↓
承認受信 → バリデーション実行 → Salesforce API実行 → 監査ログ記録
却下受信 → 却下理由ログ記録 → 担当者に「却下を受け付けました」と通知
```

## 例外ケース

### 夜間モードの顧客向けアクション

```
条件: 現在時刻が夜間モード（平日19:00〜翌9:00、または土日祝日）
対象: 顧客への外部送信（メール等）

承認済みであっても翌営業日9:00まで実行を保留する。
保留中は pending_approvals の status を "approved_waiting" とする。
翌営業日9:00以降の最初の巡回サイクルで実行する。
```

### 承認者不在（2時間タイムアウト）後のアクション

```
条件: 承認リクエスト送付から2時間以上が経過して未承認
動作: deferred_actions に移動（ブロックしない）
注意: 次サイクルのASSESSMENTで再評価する。商談状況が変化している可能性があるため、
      自動的に再承認リクエストを送付せず、ASSESSMENTで要否を再判定すること。
```

### 緊急ロールバック

```
条件: 自動実行後に誤実行が検出された場合
動作:
  1. rollback_snapshotを使用して即時ロールバック実行
  2. ロールバック完了後に担当者・管理者に通知
  3. 該当アクション種別を「一時停止」状態にする
  4. 管理者が確認・解除するまで同種アクションを承認制に格上げ
```

## 自動実行/承認制の判定フローチャート

```
アクション発生
      ↓
アクション種別が高リスクリストに含まれるか？
  YES → 承認リクエストを生成・送付
  NO  ↓
実行前チェックリストを全て満たすか？
  NO  → アクション中断・エラーログ記録
  YES ↓
rollback_snapshotを記録する
      ↓
Salesforce APIで実行
      ↓
自動実行完了通知を担当者に送付
      ↓
監査ログに記録
```

## GOOD/BAD 例

**GOOD**: ミーティング後にGmail送受信をSalesforce Activityとして自動記録した（低リスク・自動実行）。
その後、Gmailで顧客から受領した提案修正依頼に基づくフォローアップメール送信は承認リクエストとして
担当者に送付した。担当者が文面を確認・修正してから承認し、エージェントがメールを送信した（高リスク・承認制）。

**BAD**: 商談が「Proposal」ステージで15日以上停滞しているため、エージェントが「Qualification」に
ステージを後退させた（承認なしでの高リスクアクション実行）。担当者は意図しないステージ変更に
気づかず、フォーキャストレポートに誤った数値が混入した。ステージの後退は必ず承認制を経ること。
