# Output Structure Patterns — 出力テンプレート集

## 基本原則

エージェントの出力は「担当者が即座に行動できる情報密度」を目指す。
長文の説明よりも構造化されたテンプレートを優先し、
重要情報（商談名・金額・期日・推奨アクション）を先頭に置く。

---

## 1. 日次サマリー（Daily Summary）

**配信先**: チームSlackチャンネル
**配信タイミング**: 夜間モード（19:30頃）

```
📊 【営業パイプライン日次サマリー】{YYYY/MM/DD}

■ 本日の対応状況
  自動実行:   {N}件（Activity記録 {N}件、リマインダー {N}件）
  承認済み実行: {N}件
  承認待ち:   {N}件（うち期限切れ繰り延べ {N}件）
  却下:       {N}件
  検出なし:   {N}件

■ 要注意商談（クロージング期限7日以内・停滞中）
  {商談名} | ¥{金額} | クロージング: {日付}（残{N}日）| {ステージ}
  {商談名} | ¥{金額} | クロージング: {日付}（残{N}日）| {ステージ}

■ 翌日の優先タスク
  🔴 緊急: {件数}件
    - {商談名}: {アクション概要}
  🟡 重要: {件数}件
    - {商談名}: {アクション概要}

■ パイプライン概況
  アクティブ商談: {N}件（¥{合計金額}）
  加重パイプライン: ¥{加重合計金額}
  今月のパイプラインカバレッジ: {N.N}倍（目標: 3.0倍以上）

■ API使用状況
  Salesforce API使用率: {N}%（{使用数}/{上限}）
  モード: {normal / safe_mode / readonly}
```

## 2. 承認リクエスト（Approval Request）

**配信先**: 担当者Slack DM
**配信タイミング**: 高リスクアクション検出時（DISPATCHフェーズ）

```
【承認リクエスト】{検出カテゴリ} — {商談名}

■ 商談情報
  商談名:       {Opportunity名}
  金額:         ¥{金額}
  ステージ:     {現在のステージ}
  クロージング: {日付}（残{N}日）
  担当者:       {担当者名}

■ 検出内容
  {具体的な検出内容。例: 提案ステージで{N}日間更新なし。
   直近アクティビティ: {日付} メール送信（返信なし）}

■ 推奨アクション
  {具体的なアクション内容}

■ 推奨理由
  {理由}

■ 判定信頼度: {HIGH / MEDIUM / LOW}
  {LOW の場合: ※ Salesforceのデータのみに基づく判定です。
   他ソースで最近のやり取りがある場合は却下してください。}

👉 承認しますか？
  ✅ 承認（実行する）
  ❌ 却下（実行しない）
  📝 変更して実行（内容を返信してください）

⏰ {送付時刻+30分} にリマインダー送信。{送付時刻+2時間} 未承認の場合、次サイクルへ繰り延べます。
```

## 3. 自動実行通知（Auto-Execution Notification）

**配信先**: 担当者Slack DM
**配信タイミング**: 低リスクアクション自動実行後

```
✅ 【自動実行完了】{アクション種別} — {商談名}

  実行内容: {具体的な実行内容。例: GmailのActivityをSalesforceに記録しました}
  対象商談: {商談名}（{OpportunityId}）
  実行日時: {ISO8601タイムスタンプ}

  詳細: {件名 / 日時 / 相手先 等}

取り消しが必要な場合は「取り消し」と返信してください。（取り消し可能期限: 30分以内）
```

---

## 4. アクション提案（Action Suggestion）

**配信先**: 担当者Slack DM
**用途**: 承認制ではないが担当者への情報提供が必要な場合

```
💡 【アクション提案】{提案カテゴリ} — {商談名}

■ 状況
  {現在の状況説明}

■ 提案
  {具体的な提案内容}

■ 参考情報
  {Salesforce/Gmail/Calendar/Slackから取得した関連情報}

※ この提案はエージェントが自動実行しません。判断・実行は担当者にお任せします。
```

---

## 5. ロールバック通知（Rollback Notification）

**配信先**: 担当者Slack DM + 管理者チャンネル
**配信タイミング**: 自動実行のロールバック後

```
⚠️ 【ロールバック実行】{アクション種別} — {商談名}

  実行したアクション: {元のアクション内容}
  ロールバック理由: {理由}
  復元した状態:
    変更前: {before_state}
    変更後（ロールバック後）: {after_state（= before_state）}
  実行日時: {元のアクションの実行日時}
  ロールバック完了日時: {ロールバック完了日時}

Salesforceのレコードを確認してください。
ご不明な点は管理者にお問い合わせください。
```

## 6. 監査ログ（Audit Log）エントリ

**保存先**: 監査ログストレージ（30日間保持）
**記録タイミング**: 全アクション実行後（自動実行・承認済み実行・却下・繰り延べ）

```json
{
  "log_id": "{UUID}",
  "cycle_id": "{サイクルID}",
  "timestamp": "{ISO8601}",
  "event_type": "auto_executed | approval_requested | approved | rejected | deferred | rolled_back",
  "opportunity_id": "{SalesforceOpportunityId}",
  "opportunity_name": "{商談名}",
  "detection_category": "{検出カテゴリ}",
  "action_type": "{アクション種別}",
  "action_detail": "{アクション詳細}",
  "detection_confidence": "HIGH | MEDIUM | LOW",
  "approver": "{承認者のSlackユーザーID（承認制アクションのみ）}",
  "result": "{実行結果の詳細}",
  "before_state": "{変更前のSalesforceフィールド値（変更操作のみ）}",
  "after_state": "{変更後のSalesforceフィールド値（変更操作のみ）}"
}
```

**注意**: `before_state` / `after_state` にOAuthトークンや個人情報（メール本文全文）を記録しない。

## GOOD/BAD 例

**GOOD**: 日次サマリーに「要注意商談」セクションを設け、クロージング期限7日以内かつ停滞中の商談を金額・残日数付きでリストアップした。担当者は翌朝のサマリーを確認するだけで当日の優先行動が明確になった。

**BAD**: 日次サマリーを「本日{N}件の商談を確認しました。詳細はSalesforceを確認してください。」という1行にまとめた。担当者はSalesforceを開いて自分で確認する手間が発生し、エージェントの価値を感じられなかった。
