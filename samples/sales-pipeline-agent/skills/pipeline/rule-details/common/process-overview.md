# Process Overview — 営業パイプライン管理エージェント 5フェーズフロー

## フェーズ構成

営業パイプライン管理エージェントは以下の5フェーズで動作します。

```
SURVEILLANCE → ASSESSMENT → DISPATCH → REPORTING + PACKAGING
     ↑                                        |
     └────────── LOOP CHECK ──────────────────┘
                （対応件数 > 0 かつ ループ回数 < 5 かつ 経過時間 < 45分）
```

---

## Phase 1: SURVEILLANCE（4ソース巡回）

### 目的
Salesforce・Gmail・Google Calendar・Slackの4ソースを差分取得し、統合ビューを構築する。

### ステップ

**S1: スケジュールチェック**
- 現在時刻が日中モード（平日9:00〜19:00）か夜間モードかを判定
- Salesforce API使用量を確認（使用率 ≥ 80%: セーフモード移行、≥ 95%: 読み取り専用モード）

**S2: ソース接続確認**
- 4ソースのOAuthトークン有効期限を確認
- 有効期限30分以内のトークンをリフレッシュ
- 接続失敗ソースはフォールバック処理へ（部分巡回続行）

**S3: データ取得（差分）**
- Salesforce: 前回巡回以降に更新された商談・リード・Activityを取得
- Gmail: 前回巡回タイムスタンプ以降の送受信メールを取得
- Google Calendar: 過去7日〜今後14日の外部参加者を含むミーティングを取得
- Slack: 前回巡回以降の監視対象チャンネルのメッセージ・キーワードを取得

**S4: 統合ビュー構築**
- 商談単位でSalesforce/Gmail/Calendar/Slackの情報を紐づけ
- タイムスタンプベースで最新情報を優先
- ソース間矛盾を検出しフラグを付与（Pitfall 5対策）

### 出力
統合ビュー（商談ごとのSalesforce状態 + マルチソースアクティビティタイムライン）

---

## Phase 2: ASSESSMENT（対応要否判定）

### 目的
統合ビューを基に、対応が必要な商談・リードを多層判定で特定する。

### 判定ロジック

**停滞商談の多層判定**（Pitfall 5対策 — 単一ソースでの判定禁止）

```
Step 1: Salesforce停滞チェック
  同一ステージの滞留日数 >= 閾値（デフォルト: 14日）
        ↓
Step 2: マルチソースアクティビティ確認
  直近7日以内にGmail/Calendar/Slackでアクティビティがあるか？
  → YES: 停滞スコアを下方修正（Salesforce更新漏れの可能性）
  → NO:  停滞スコアを確定
        ↓
Step 3: ヘルススコア算出（0-100）
  滞留日数スコア + エンゲージメントスコア + 期限リスクスコアの複合評価
```

**その他の判定カテゴリ**

| 判定 | 条件 | 閾値 |
|------|------|------|
| 未返信メール | 送信後返信なし | 3日/7日/14日の段階的アラート |
| ミーティング準備 | カレンダー予定 + 準備サマリー未生成 | ミーティング24時間前 |
| クロージング期限接近 | クロージング日まで残日数が少なく停滞中 | 7日以内 |
| アクティビティ未記録 | Gmail/Calendar/SlackイベントがSalesforce Activityに未記録 | 即時検出 |
| データ品質劣化 | 必須フィールド未入力 / 最終更新が古い | 30日以上未更新 |
| 未追跡リード | 新規メール/Slack言及でSalesforce未登録の見込み顧客 | 即時検出 |

### 出力
対応アクションリスト（アクション種別・対象商談・優先度・推奨内容）

---

## Phase 3: DISPATCH（アクション実行）

### 目的
対応アクションリストを処理する。低リスクは自動実行、高リスクは承認リクエストを送付する。

### 実行フロー

```
対応アクションリスト受け取り
        ↓
アクションごとにリスク分類
        ↓
[低リスク]                    [高リスク]
  ↓                              ↓
自動実行                      承認リクエスト生成
  ↓                              ↓
Salesforce API実行            Slack DM 送付
  ↓                              ↓
実行ログ記録                  承認/却下待機（最大2時間）
                                  ↓
                             承認 → 実行 → ログ記録
                             却下 → 却下理由記録
                           2時間タイムアウト → 次サイクルへ繰り延べ
```

### 並行実行の注意事項
- 同一商談への複数アクションは逐次実行（排他制御）
- 異なる商談へのアクションは並行実行可
- Salesforce DML操作の1トランザクション150件上限を遵守

---

## Phase 4: REPORTING（結果レポート）

### 目的
巡回サイクルの処理結果を集約し、担当者・チームに配信する。

### レポート種別

| レポート | 配信先 | タイミング |
|---------|--------|-----------|
| 巡回サマリー（簡易） | 担当者Slack DM | 各サイクル完了後 |
| 日次ダイジェスト | チームSlackチャンネル | 夜間モード（19:30） |
| 週次パイプラインレビュー素材 | チームSlackチャンネル | 金曜夜間モード |

### 監査ログ記録
全アクション（自動実行・承認制・却下）をログに記録する。ログ項目:
- タイムスタンプ
- 検出カテゴリ
- 対象商談/リードID
- 推奨アクション
- 実行結果（自動実行 / 承認済み実行 / 却下 / 繰り延べ）

---

## Phase 5: PACKAGING（繰り延べ処理）

### 目的
現サイクルで完了できなかったアクションを次サイクルへ安全に引き継ぐ。

### 繰り延べ発生条件
- 45分タイムアウト到達
- 最大5ループ到達後に残件あり
- 承認待ち2時間タイムアウト

### 繰り延べ処理
```
繰り延べリストを生成
  ↓
各アクションに「繰り延べ理由」と「繰り延べ回数」を付与
  ↓
次サイクルのASSESSMENTフェーズに先頭で投入
  ↓
繰り延べ3回以上のアクション → 管理者アラート
```

---

## ループ制御

| 制御項目 | 設定値 | 超過時の動作 |
|---------|-------|------------|
| 最大ループ回数 | 5回/サイクル | 残件を繰り延べ処理 |
| サイクルタイムアウト | 45分 | 強制終了→繰り延べ処理 |
| 承認待ちタイムアウト | 2時間 | 次サイクルへ繰り延べ |
| 承認リマインダー | 30分後 | 担当者Slack DM再送 |

---

## GOOD/BAD 例

**GOOD**: ASSESSMENT フェーズで、Salesforceでは20日間ステージが止まっている商談を検出した。
Slackを確認すると前日に活発なやり取りがあったため、停滞スコアを下方修正し「更新リマインダー」のみを送付した。
Salesforce未更新（Pitfall 1）を誤って停滞商談と判定することを防いだ。

**BAD**: ASSESSMENTフェーズで複数の商談が停滞と判定され、DISPATCHフェーズで承認リクエストを5件同時送付した。
担当者が通知に圧倒され、重要なクロージング期限接近アラートを見逃した（Pitfall 3: 通知疲れ）。
承認リクエストは優先度順に分割送付し、緊急案件を先頭に置くべきだった。
