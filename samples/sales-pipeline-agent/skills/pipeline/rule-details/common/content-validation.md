# Content Validation — Salesforceデータバリデーション・メール内容チェック・金額整合性検証

## 基本原則

エージェントが読み取り・書き込みするすべてのデータに対してバリデーションを実施する。
特にSalesforceへの書き込み操作（DML）は、実行前に必ず検証を完了させる。
バリデーション失敗時は操作を中断し、エラーログを記録した上で担当者に通知する。

---

## 1. Salesforceレコードバリデーション

### 1-1. 商談（Opportunity）の必須フィールド検証

```
バリデーション対象フィールド:
  - Name（商談名）: 必須、文字列、255文字以内
  - StageName（ステージ）: 必須、許可値リストに含まれること
  - CloseDate（クロージング日）: 必須、日付形式、過去日でないこと（新規作成時）
  - Amount（金額）: 任意だが入力時は数値 >= 0
  - OwnerId（担当者）: 必須、有効なSalesforceユーザーIDであること
  - AccountId（取引先）: 推奨、商談化後は必須
```

**ステージ許可値リスト**（設定に応じて変更可）
```
許可値: Prospecting, Qualification, Proposal, Negotiation,
       Closed Won, Closed Lost
遷移ルール:
  - 前進更新（自動実行可）: Prospecting→Qualification, Qualification→Proposal 等
  - 後退更新（承認制）: Proposal→Qualification 等、前のステージへの変更
  - クローズ更新（承認制）: 任意ステージ → Closed Won/Lost
```

### 1-2. リード（Lead）の必須フィールド検証

```
バリデーション対象フィールド:
  - LastName（姓）: 必須
  - Company（会社名）: 必須
  - Email: 任意だが入力時は正規表現でフォーマット検証
  - Status（リードステータス）: 許可値リストに含まれること
```

### 1-3. Activity（活動）の登録バリデーション

```
バリデーション対象フィールド:
  - Subject（件名）: 必須、文字列
  - ActivityDate（活動日）: 必須、日付形式
  - WhatId（関連商談/取引先ID）: 必須、対象オブジェクトが存在すること
  - WhoId（コンタクト/リードID）: 推奨
  - Description（説明）: 任意、32,768文字以内
```

### 1-4. SOQL クエリバリデーション

```
禁止パターン:
  - SELECT * FROM ... （全フィールド取得禁止 → ガバナーリミット対策）
  - WHERE句なしの全件取得（50,000行上限超過リスク）
  - LIMIT句なし（大量データ取得リスク）

推奨パターン:
  - SELECT Id, Name, StageName, CloseDate, Amount FROM Opportunity
    WHERE LastModifiedDate >= :lastSyncTimestamp
    ORDER BY LastModifiedDate DESC
    LIMIT 200
```

## 2. メール内容チェック

承認を受けた顧客向けメール送信前に以下のチェックを実施する。

### 2-1. 送信前必須チェック

```
チェック項目:
  □ 宛先（To）: Salesforceコンタクト/リードのメールアドレスと一致するか
  □ 件名: 空でないか、スパムフィルターに引っかかる可能性のある文字列を含まないか
  □ 本文: 空でないか、プレースホルダー（{名前}、{日付}等）が未置換のまま残っていないか
  □ 商談名・金額: 本文内の商談名がSalesforceレコードの商談名と一致するか
  □ 金額表記: 本文内の金額がSalesforceの金額フィールドと一致するか（±10%以内）
  □ 添付ファイル: 意図しない添付がないか
```

### 2-2. プレースホルダー未置換チェック

```
検出パターン（正規表現）:
  - \{[A-Za-z一-龥ぁ-ん]+\}  （{顧客名} 等の未置換プレースホルダー）
  - \[.*?\]                    （[XXX] 等のブラケット形式のプレースホルダー）
  - TODO|FIXME|TBD             （ドラフト残留マーカー）

検出時の動作:
  → メール送信を中断
  → エラー内容を担当者に通知（「本文に未置換のプレースホルダーが含まれています: {X}」）
  → 修正後に再承認を要求
```

### 2-3. 送信対象の二重確認

```
高リスク判定条件（いずれか一つ該当で追加確認を実施）:
  - 金額 >= ¥10,000,000（1千万円以上の案件）
  - 宛先が新規コンタクト（Activityログが3件未満）
  - 件名に「最終」「クローズ」「解約」等の重要ワードを含む
```

## 3. 金額整合性検証

商談金額に関連するデータ操作前に整合性を検証する。

### 3-1. 金額変更バリデーション

```
変更前チェック:
  - 変更前金額（Salesforce現在値）を取得して記録（ロールバック用）
  - 変更後金額 >= 0 であること
  - 変更後金額が変更前金額の50%未満または200%超の場合 → 異常変動フラグ

異常変動時の動作:
  → 自動実行を中断（金額変更は承認制のため、そもそも承認なしでは実行しない）
  → 承認リクエストに「変更前金額: ¥X → 変更後金額: ¥Y（変動率: Z%）」を明示
  → 担当者の確認を必須とする
```

### 3-2. フォーキャスト整合性チェック

```
チェック条件:
  - 個別商談の金額変更後、加重パイプライン合計が前回レポート値から20%以上乖離する場合
    → 月次フォーキャストへの影響サマリーを生成し、マネージャーに通知

加重パイプライン算出:
  加重パイプライン = Σ（商談金額 × ステージ別受注確率）

  ステージ別受注確率（デフォルト値、設定変更可）:
    Prospecting:   10%
    Qualification: 20%
    Proposal:      40%
    Negotiation:   60%
    Closed Won:   100%
    Closed Lost:    0%
```

### 3-3. 重複金額検出

```
チェック条件:
  - 同一企業（AccountId）に対して金額・商談名が類似した複数の商談が存在する場合
    → 重複候補として検出
  - 類似判定基準: 商談名の類似度 >= 80% かつ 金額差が ±20% 以内
  → 検出時は担当者に確認依頼（自動マージは禁止）
```

## 4. APIレスポンス検証

Salesforce APIレスポンスに対して以下の検証を実施する。

```
成功レスポンス確認:
  - HTTP 200/201: 正常処理
  - DML操作後: success: true、id フィールドが返ること

エラーレスポンス対応:
  - HTTP 400（MALFORMED_QUERY）: SOQLクエリを修正して1回リトライ
  - HTTP 401（INVALID_SESSION_ID）: トークンリフレッシュ後に1回リトライ
  - HTTP 403（REQUEST_LIMIT_EXCEEDED）: セーフモード移行・管理者アラート
  - HTTP 500（SERVER_ERROR）: 指数バックオフでリトライ（最大3回）
  - それ以外: エラーログ記録・担当者通知・操作中断
```

## GOOD/BAD 例

**GOOD**: 承認済みのフォローアップメールを送信する前に、本文内のプレースホルダー検証を実施した。
「{顧客名}様」が未置換のまま残っていることを検出し、メール送信を中断して担当者に修正を依頼した。
顧客への失礼なメール送信を未然に防いだ。

**BAD**: 商談金額を¥5Mから¥50Mに変更するSOQLクエリを実行した際、変更前金額の記録を取らずに実行した。
その後誤変更と判明したが、元の金額を復元できずに手動確認が必要になった。
金額変更操作は変更前の状態を必ず記録（ロールバック用）してから実行すること。
