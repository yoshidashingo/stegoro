# Overconfidence Prevention — Salesforceデータの過信防止・自動実行の保守的判定

## 基本原則

Salesforceのデータは「現実の商談状況の近似値」であり、「絶対的な事実」ではない。
エージェントは常に「Salesforceの情報が古い可能性」「データ入力漏れの可能性」を
前提として判定を行う。

特に以下の判定には過信を防ぐ制約を設ける。

---

## 1. 停滞判定の過信防止

### 禁止: 単一指標による停滞判定

```
禁止パターン:
  StageName == "Proposal" かつ LastModifiedDate < 14日前
  → これだけで「停滞商談」と断定して承認リクエストを送付する

理由:
  - 営業担当がSalesforceを更新していないだけで、実際は進行中の可能性がある
  - Slackや社内会議で商談が動いていてもSalesforceに反映されていないケースが多い
  - 単一指標での判定はPitfall 1（CRMデータ鮮度劣化）と Pitfall 5（ソース間不整合）の
    直接原因となる
```

### 必須: 多層判定による停滞確認

```
停滞判定に必要な条件（全条件を確認すること）:

  条件1: Salesforce停滞チェック
    → StageName が同一で LastModifiedDate >= N日前（閾値: 14日）

  条件2: Gmailアクティビティ確認
    → 直近7日以内に該当商談の取引先ドメインとのメール送受信がないこと

  条件3: Calendarアクティビティ確認
    → 直近7日以内に外部参加者を含むミーティングがないこと

  条件4: Slackアクティビティ確認
    → 直近7日以内に監視チャンネルで商談名・顧客名の言及がないこと

判定結果:
  条件1のみ該当: 「SF未更新の可能性あり」→ 更新リマインダーのみ送付
  条件1-2該当:   「停滞の可能性中程度」→ 担当者への確認依頼
  条件1-4全該当: 「停滞と判定」→ フォローアップ承認リクエスト送付
```

## 2. 自動実行の保守的判定

### 自動実行前の必須チェック

```
以下のいずれかに該当する場合、自動実行を中止して承認制に格上げする:

  □ 同一商談に対して過去24時間以内に自動実行を行っている
  □ 変更対象のフィールドが「高リスクフィールドリスト」に含まれる
     （Amount, StageName の後退, CloseDate の変更, IsClosed）
  □ 対象商談のクロージング日まで7日以内（緊急フラグ — 人間判断を優先）
  □ 商談金額が¥10,000,000以上（大型案件 — 慎重な扱いが必要）
  □ 対象のActivity種別が「顧客への外部連絡」を含む
  □ Salesforceレコードに矛盾フラグが付与されている
```

### 低リスク自動実行でも確認すること

低リスクアクション（Activityログ記録等）であっても、以下を確認してから実行する。

```
事前確認チェックリスト:
  □ 対象のOpportunityIdが実在するか（SOQL SELECT で事前確認）
  □ 同一内容のActivityが直近24時間以内に既に記録されていないか（重複登録防止）
  □ WhoId（コンタクトID）が実在するか
  □ ActivityDateが未来日でないか（誤日付防止）
```

## 3. 商談金額・ステージ変更への慎重な姿勢

### 金額変更

```
エージェントは商談金額を能動的に変更しない。

許可される金額操作:
  - Gmail/Slackから読み取った金額をレポートに記載する（参照のみ）
  - 金額の不整合を検出して担当者に確認依頼する

禁止される金額操作:
  - 承認なしでSalesforceのAmountフィールドを変更する
  - メール本文の金額を参照してSalesforceを自動更新する
  - 「おそらくこの金額だ」という推測に基づく更新

理由:
  フォーキャスト・売上報告・コミッション計算に直接影響するため、
  わずかな誤更新が経営判断を誤らせる可能性がある。
```

### ステージ後退の禁止（自動実行）

```
ステージ後退（例: Proposal → Qualification）は絶対に自動実行しない。

理由:
  - 担当者のモチベーション・商談履歴・パイプライン予測に影響する
  - 後退の判断には商談の文脈（交渉経緯・顧客の意思決定プロセス）が必要
  - エージェントが持つデータのみでは文脈を完全に把握できない

ステージ後退が必要と判断した場合:
  → 承認リクエストとして「ステージ後退の提案」を担当者に送付する
  → 「提案」であることを明記し、担当者の最終判断に委ねる
```

## 4. 判定信頼度の明示

承認リクエストに判定信頼度を付記する。

```
判定信頼度ラベル:
  HIGH:   4ソース全てのデータが一致して停滞を示している
  MEDIUM: 3ソースが停滞を示し、1ソースは確認できなかった
  LOW:    Salesforceのみが停滞を示しており、他ソースは確認できない

LOW判定時は承認リクエストに以下を付記する:
「※ Salesforceのデータのみに基づく判定です。
   Gmail/Slack/Calendarで最近のやり取りがある場合は却下してください。」
```

## GOOD/BAD 例

**GOOD**: 商談Aが提案ステージで20日間停滞とSalesforceが示していた。
エージェントはGmailを確認したところ、3日前に顧客から返信があることを検出した。
「停滞判定: LOW（Salesforceのみ）。Gmailでは直近にやり取りあり」として更新リマインダーのみを送付し、
フォローアップ承認リクエストの送付を回避した。

**BAD**: Salesforceのステージ滞留日数が15日を超えたため、他ソースを確認せずに
「停滞商談」と断定してフォローアップメール送信の承認リクエストを送付した。
実際には担当者がSlackで顧客と毎日やり取りしており、エージェントからの割り込みで
交渉が混乱した（Pitfall 1・Pitfall 5）。
