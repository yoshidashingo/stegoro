# S2: Source Connection — Phase Rule

## Purpose

4データソース（Salesforce / Gmail / Google Calendar / Slack）への接続確認とOAuth認証を実行する。アクセストークンの有効期限チェックとリフレッシュ、接続失敗時の部分巡回フォールバック判定を行い、後続のデータ収集（S3）が利用可能なソース一覧を確定する。

**Classification**: ALWAYS EXECUTE（日中モード時に必須）
**Approval Gate**: なし（次ステージ S3 へ自動移行）
**参照元**: `workflow-architecture.md` S2 セクション

---

## Prerequisites

- S1で日中モードと判定されていること（夜間モードではR2へジャンプ済みのためS2は実行されない）
- S1のサイクル状態ファイルが初期化されていること
- 4ソースのOAuth認証情報が `config/data-source-config.md` に設定済みであること
- S1のAPIモード判定結果（通常 / セーフ / 読み取り専用）が確定していること

---

## Execution Classification

**ALWAYS EXECUTE**（日中モード時）

- Execute IF: S1で日中モード判定の場合
- Skip IF: S1で夜間モード判定の場合（このステージには到達しない）

---

## Execution Steps

### Step 1: 4ソースの並行接続確認

4ソースへの接続確認を **並行実行** する（直列実行禁止。合計タイムアウトを抑えるため）。

各ソースの接続確認項目:

| ソース | 接続確認方法 | タイムアウト |
|--------|------------|------------|
| Salesforce | `/services/data/v58.0/` エンドポイントへのヘルスチェックGET | 10秒 |
| Gmail | `gmail.users.getProfile` APIコール（メタデータのみ） | 10秒 |
| Google Calendar | `calendar.calendarList.list` APIコール（一覧取得） | 10秒 |
| Slack | `auth.test` APIメソッド | 10秒 |

**接続結果の記録**:
```yaml
source_status:
  salesforce: connected      # connected / auth_error / rate_limit / timeout / partial
  gmail: connected
  google_calendar: connected
  slack: auth_error
```

### Step 2: OAuthアクセストークンの有効期限チェックとリフレッシュ

各ソースのアクセストークン有効期限を確認し、必要に応じてリフレッシュを実行する。

**リフレッシュ判定ルール**:
- トークン残り有効期間 > 30分: リフレッシュ不要
- トークン残り有効期間 <= 30分: リフレッシュトークンで自動更新
- リフレッシュトークン自体が失効: `auth_error` として記録

**各ソースのOAuth仕様**:

| ソース | アクセストークン有効期間 | リフレッシュ方法 |
|--------|----------------------|----------------|
| Salesforce | 設定依存（通常2〜12時間） | OAuth 2.0 refresh_token grant |
| Gmail | 1時間 | Google OAuth 2.0 token endpoint |
| Google Calendar | 1時間（Gmailと同一トークン） | Google OAuth 2.0 token endpoint |
| Slack | 無期限（ユーザートークン）/ 12時間（Bot token） | OAuth 2.0 refresh_token grant |

**リフレッシュ成功時**: 新トークンを `config/oauth-tokens.json` に安全に保存（上書き）
**リフレッシュ失敗時**: 当該ソースを `auth_error` フラグで記録し、Step 3のフォールバック判定へ

### Step 3: 部分障害フォールバック判定

接続・認証に失敗したソースを除いて巡回を続行するか判定する。

**フォールバック判定マトリクス**:

| 障害ソース | 対処 | 影響するS4検出カテゴリ |
|-----------|------|----------------------|
| Salesforceのみ障害 | 読み取り専用続行（承認制アクションの実行を停止） | 全カテゴリに影響（Salesforceは基軸データ）。検出は実施するが書き込み系アクションは保留 |
| Gmailのみ障害 | Gmail由来カテゴリをスキップして続行 | 未返信メール検出、ステータス未反映（Gmail由来）をスキップ |
| Google Calendarのみ障害 | Calendar由来カテゴリをスキップして続行 | 会議準備検出をスキップ |
| Slackのみ障害 | Slack由来カテゴリをスキップして続行 | 停滞判定の多層チェック（Slack部分）、ステータス未反映（Slack由来）をスキップ |
| 2〜3ソース障害 | 利用可能ソースのみで続行。障害内容をレポートに記録 | 各障害ソースの検出カテゴリをスキップ |
| 全ソース（4ソース）障害 | 巡回をスキップ。管理者緊急通知を送信してR3（Audit Log）へジャンプ | 全カテゴリスキップ |

**Salesforce障害時の特別処置**:
Salesforceは商談データの基軸となるため、Salesforceが障害の場合でも他3ソースでのデータ収集は実施する。ただし:
- S4の検出結果は「参考情報」扱いとし、自動実行アクションを生成しない
- 管理者へ「Salesforce障害のため今サイクルは読み取り参照のみ」とSlack通知

### Step 4: 接続状態のサイクル状態ファイルへの記録

Step 3の判定結果をサイクル状態ファイルに記録する。

```markdown
## Source Status
- **Salesforce**: connected
- **Gmail**: auth_error（トークンリフレッシュ失敗）
- **Google Calendar**: connected
- **Slack**: connected

## Skipped Detection Categories（S4）
- 未返信メール検出: スキップ（Gmail障害）
- ステータス未反映（Gmail由来）: スキップ（Gmail障害）
```

S3（Data Collection）はこのファイルを参照し、利用可能なソースのみからデータ収集を実施する。

---

## Output Artifacts

S2 が生成・更新するアーティファクト:

1. **ソース接続状態**（サイクル状態ファイル内） — S3のデータ収集範囲制御に使用
2. **スキップカテゴリリスト** — S4の検出ルール適用範囲の制限に使用
3. **更新済みOAuthトークン**（`config/oauth-tokens.json`） — S3以降のAPI呼び出しに使用
4. **障害通知ログ** — 障害ソースが存在する場合、R1のサマリーに「障害ソースあり」として記録


## Error Handling

### エラー 1: 接続失敗を理由とした巡回全停止（Pitfall 4対策）

**症状**: Salesforce接続に失敗した際、他3ソースで取得可能なデータがあるにもかかわらず巡回全体が停止する。
**原因**: Step 3のフォールバック判定を実装せず、いずれかのソース障害で即座に例外をスローする設計。
**対処**: 各ソースの接続失敗は個別に処理し、利用可能なソースでの部分巡回を続行する。全ソース障害時のみ巡回をスキップする。

**GOOD例**:
```
Gmailのアクセストークンが期限切れ → リフレッシュ試行
→ リフレッシュ成功: 新トークン取得、Gmail接続を再確立
→ 全4ソース利用可能でS3へ続行
（自動リフレッシュによる中断なし）
```

**BAD例**:
```
Salesforce接続タイムアウト（10秒）
→ 例外をスローして巡回全体を中断
→ Gmail・Calendar・Slackで取得可能なデータが失われる
→ 当日の会議準備サマリーが生成されず、担当者が準備不足でミーティングに臨む
（部分続行フォールバックなし）
```

### エラー 2: トークンリフレッシュ失敗の無視（Pitfall 4対策）

**症状**: リフレッシュトークンが失効しているにもかかわらず、エラーを無視して期限切れトークンでAPI呼び出しを継続し、401エラーが多発する。
**原因**: リフレッシュ失敗時の `auth_error` フラグ記録とS3への通知が未実装。
**対処**: リフレッシュ失敗は `auth_error` として明示的に記録し、S3ではそのソースをスキップする。管理者にOAuth再認証が必要である旨を通知する。

**GOOD例**:
```
Gmailリフレッシュトークン失効を検出
→ auth_error フラグを記録
→ 管理者Slack通知: 「Gmail OAuth再認証が必要です。巡回は他3ソースで続行します」
→ S3でGmailデータ収集をスキップ
→ 未返信メール検出カテゴリをスキップとR1サマリーに明記
（障害の透明性確保と部分続行）
```

**BAD例**:
```
Gmailリフレッシュトークン失効
→ エラーを無視して期限切れトークンでGmail API呼び出し
→ 401 Unauthorized エラーが連続発生
→ API呼び出しリトライが繰り返されSalesforce API消費を増大（Pitfall 4波及）
→ ガバナーリミットが枯渇し全ソースの巡回が中断
```

### エラー 3: 直列実行によるタイムアウト超過

**症状**: 4ソースの接続確認を直列実行したため、各10秒 × 4 = 40秒かかり、45分サイクルタイムアウトの消費が増大する。
**原因**: Step 1の並行実行指示を無視した直列実装。
**対処**: 4ソースの接続確認は必ず並行実行する。最大タイムアウトは並行実行で10秒（直列40秒比較で30秒削減）。

**GOOD例**:
```
4ソース並行接続確認開始
→ Salesforce: 3秒で応答
→ Gmail: 2秒で応答
→ Google Calendar: 4秒で応答
→ Slack: 1秒で応答
→ 合計: 4秒（最遅のCalendarに合わせる）
→ サイクル残り時間への影響を最小化
```

**BAD例**:
```
4ソース直列接続確認
→ Salesforce: 8秒 → Gmail: 9秒 → Calendar: 7秒 → Slack: 6秒
→ 合計: 30秒消費
→ 45分サイクルの約1%を接続確認だけで消費
→ タイムアウトリスクが増大
```

### エラー 4: 全ソース障害時の無限リトライ（Pitfall 4対策）

**症状**: 全4ソースが障害状態にもかかわらず、接続リトライを繰り返してサイクル全体がタイムアウトする。
**原因**: 全ソース障害時のアーリーエグジット処理が未実装。
**対処**: 全ソース障害を検出した時点で即座に管理者通知を送信し、R3（Audit Log）へジャンプする。リトライは次サイクル開始時に自動的に実施される。

**GOOD例**:
```
4ソース全て接続失敗
→ 「全ソース障害」として判定
→ 管理者緊急通知（Slack #sales-pipeline-ops）を即時送信
→ R3（Audit Log Finalization）へジャンプ
→ 障害記録を監査ログに保存してサイクル終了
→ 次サイクル（1時間後）で自動リトライ
```

**BAD例**:
```
4ソース全て接続失敗
→ 各ソースで3回リトライ × 4ソース = 12回のリトライ実行
→ 各リトライ10秒タイムアウト = 120秒消費
→ その後も処理を続けようとしてサイクルタイムアウト
→ 管理者に通知が届かず、障害に気づくのが遅れる
```
