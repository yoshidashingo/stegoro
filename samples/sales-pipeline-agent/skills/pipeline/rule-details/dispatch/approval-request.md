# D2: Approval Request — Phase Rule

## Purpose

A4の承認制バッチリストに含まれる高リスクアクションの承認リクエストを送信する。コンテキスト付きの構造化メッセージを担当者Slack DM（プライマリ）またはメール（フォールバック）で送信し、各リクエストに一意のリクエストIDを発行してD3での結果照合を可能にする。夜間モード中の顧客向けアクションは承認リクエストは送信するが実行は翌8:00まで保留する。

**Classification**: CONDITIONAL（承認制対象がある場合）
**Approval Gate**: なし（次ステージ D3 または D4 へ自動移行）
**参照元**: `workflow-architecture.md` D2 セクション

---

## Prerequisites

- A4の承認制バッチリスト（`state/approval-batch-list.yaml`）が確定していること
- S2でSlack接続が `connected` 状態であること（フォールバック: Gmail）
- S1の時間帯モード（日中 / 夜間）が確定していること
- 承認リクエストの送信済み記録（`state/approval-requests.yaml`）が参照可能であること

---

## Execution Classification

**CONDITIONAL EXECUTE**

- Execute IF: A4の承認制リストにアクションが **1件以上** ある場合
- Skip IF: 承認制対象が0件の場合 → D4（Loop Control）へ直行

---

## Execution Steps

### Step 1: リクエストIDの発行と送信準備

各承認バッチにリクエストIDを発行し、送信準備を行う。

**リクエストID発行規則**: `APPR-{YYYYMMDD}-{3桁連番}`

```yaml
approval_request:
  request_id: "APPR-20260405-001"
  batch_id: "BATCH-U123456-001"
  recipient_slack_id: "U123456789"
  recipient_name: "田中 太郎"
  sent_at: null  # 送信後に記録
  status: "pending"  # pending / approved / rejected / timed_out / deferred
  reminder_sent_at: null  # 30分後リマインダー送信時刻
  deferred_at: null       # 2時間後繰り延べ時刻
  actions:
    - action_id: "ACT-20260405-003"
      priority: P1
```

### Step 2: 承認リクエストメッセージの生成

バッチ内の各アクションをコンテキスト付きで1通のメッセージにまとめる。

**メッセージフォーマット（Slack Block Kit）**:

```
【承認リクエスト】APPR-20260405-001

━━━━━━━━━━━━━━━━━━━━━━
📋 アクション 1 / 3（P1 — Critical）
━━━━━━━━━━━━━━━━━━━━━━
対象商談: 株式会社H — Enterpriseプラン導入
推奨アクション: フォローアップメール送信
理由: 「提案書ご確認のお願い」への返信が3営業日（4/1送信）ありません

商談詳細:
  金額: ¥12,000,000（大型商談）
  ステージ: Proposal / クロージング予定: 2026-04-06（明日）
  ヘルススコア: 赤（35点）

下書きメール内容:
  宛先: yamada@client-h.co.jp
  件名: Re: 提案書ご確認のお願い
  本文: 田中様、先日ご送付した提案書につきまして...（省略）

━━━━━━━━━━━━━━━━━━━━━━
📋 アクション 2 / 3（P2）
━━━━━━━━━━━━━━━━━━━━━━
対象商談: 株式会社F — SaaS基盤移行
推奨アクション: ステージ更新提案（Proposal → Negotiation）
理由: 顧客から価格交渉の返信あり（4/4）、次フェーズへの進行が適切と判断

━━━━━━━━━━━━━━━━━━━━━━

⏰ このリクエストは2時間後（12:10）に自動繰り延べされます
30分後（10:40）にリマインダーを送信します

[✅ 全件承認] [❌ 全件却下] [📝 個別に対応]
```

**個別対応ボタン押下時**: 各アクションに対して `[承認] [却下（理由入力）]` ボタンを展開

### Step 3: 夜間モード中の顧客向けアクション保留フラグ付与

S1で夜間モードと判定されている場合、顧客向け送信アクションに保留フラグを付与する。

**保留対象アクション**: `send_email`（顧客向け）
**保留解除条件**: 翌営業日 8:00（JST）

```yaml
# send_email アクションへの保留フラグ付与
deferred_execution:
  reason: "nighttime_mode"
  execute_after: "2026-04-06T08:00:00+09:00"
```

**承認リクエストメッセージへの付記**:
```
⚠️ このアクションは現在夜間モードです。
承認後も顧客へのメール送信は翌営業日 2026/4/6 8:00 以降に実行されます。
```

夜間モード中でも承認リクエスト自体は送信する（担当者が翌朝前に承認準備できるよう）。

### Step 4: Slack DM 送信（プライマリ）

`chat.postMessage` APIで担当者Slack DMに送信する。

```json
POST https://slack.com/api/chat.postMessage
{
  "channel": "{recipient_slack_id}",
  "blocks": [{承認リクエストBlock Kit JSON}],
  "text": "【承認リクエスト】{request_id} — {件数}件のアクションの承認をお願いします"
}
```

**送信成功時**:
- `sent_at` を現在時刻に更新
- `status: "pending"` に設定
- 30分後リマインダーの送信スケジュールをサイクル状態ファイルに記録

**送信失敗時（SlackAPIエラー・接続不可）**:
- メールフォールバックへ移行（Step 5）

### Step 5: メール送信フォールバック

Slack送信が失敗した場合、担当者のメールアドレスへ送信する。

```
宛先: {担当者メールアドレス}（config/user-settings.yaml 参照）
件名: 【承認リクエスト】APPR-20260405-001 — {件数}件の対応確認
本文: Slack送信に失敗したためメールでお送りしています。
      {アクション詳細をテキスト形式で記載}
      承認/却下はこちら: {承認用URL または折り返し連絡先}
```

**両方失敗した場合**:
- 承認リクエスト送信失敗として記録
- 対象アクションを「次サイクル繰り延べ」に移動
- 管理者Slack通知（`#sales-pipeline-ops`）に「承認リクエスト送信失敗」を報告

### Step 6: タイムアウト管理スケジュールの登録

送信完了後、D4のタイムアウト管理用スケジュールをサイクル状態ファイルに記録する。

```yaml
approval_timeout_schedule:
  - request_id: "APPR-20260405-001"
    sent_at: "2026-04-05T10:10:00+09:00"
    reminder_due_at: "2026-04-05T10:40:00+09:00"  # 30分後
    deferred_due_at: "2026-04-05T12:10:00+09:00"  # 2時間後
    status: "pending"
```

D4（Loop Control）はこのスケジュールを参照してリマインダー送信と繰り延べを管理する。

## Output Artifacts

D2 が生成するアーティファクト:

1. **承認リクエスト送信記録**（`state/approval-requests.yaml`） — D3・D4で参照
2. **タイムアウトスケジュール** — サイクル状態ファイルに追記、D4で参照
3. **夜間保留フラグ** — D3の承認処理で参照

## Completion Message

D2 完了時のメッセージ形式は以下の通り。

```
### REVIEW REQUIRED — D2: Approval Request 完了

送信結果:
- 送信成功（Slack）: [X]件
- 送信成功（メールフォールバック）: [X]件
- 送信失敗（繰り延べ）: [X]件

夜間モード保留フラグ: [X]件（翌朝8:00以降に実行）
タイムアウトスケジュール登録: [X]件（30分リマインダー / 2時間繰り延べ）

### WHAT'S NEXT
**A) 修正依頼** — 承認リクエストのフォーマットや送信設定を修正する
**B) [D3 Approval Processing]へ続行** — 承認結果の処理へ進む（結果待ち）
**C) [D4 Loop Control]へ続行** — 承認結果なし、残件チェックへ
```

## Error Handling

### エラー 1: 承認リクエストの大量個別送信（Pitfall 3対策）

**症状**: 1担当者へ10件の承認リクエストを1件ずつSlack DM送信し、担当者のDMが承認リクエストで埋め尽くされる。担当者が全件を無視するようになる。
**原因**: A4でのバッチ化が未実施、またはD2でバッチを分解して個別送信している。
**対処**: D2はA4から受け取った「バッチ単位」で送信する。バッチ内のアクションを個別メッセージに分解することは禁止。

**GOOD例**:
```
担当者 田中太郎宛 承認バッチ: 3件（BATCH-U123456-001）
→ 1通のSlackメッセージに3件をまとめて送信
→ 担当者が受け取るメッセージ: 1件
→ [全件承認] [全件却下] [個別に対応] ボタンで効率的に処理可能（Pitfall 3対策）
```

**BAD例**:
```
BATCH-U123456-001（3件）をバッチ分解
→ ACT-003: 1件目のSlack DM送信
→ ACT-007: 2件目のSlack DM送信
→ ACT-011: 3件目のSlack DM送信
→ 担当者のDMに3件の個別メッセージ
→ バッチ内に10件あれば10件の個別メッセージが届く（Pitfall 3対策漏れ）
```

### エラー 2: 夜間モード中の顧客向けメール即時実行（ドメイン固有）

**症状**: 承認リクエストを送信し、夜間に担当者が承認すると顧客への送信が深夜に実行される。
**原因**: Step 3の夜間モード保留フラグ付与が未実装。
**対処**: `send_email` アクションに必ず `deferred_execution.execute_after` を設定し、D3での実行時に時刻チェックを実施する。

**GOOD例**:
```
現在時刻: 夜間モード（22:00）
→ send_email アクション: deferred_execution.execute_after = "2026-04-06T08:00:00+09:00"
→ 承認リクエスト送信（承認準備のみ）
→ 担当者が23:00に承認
→ D3: 承認確認 → execute_after チェック: 08:00 未到達
→ 送信キューに登録、翌朝8:00に実行
→ 顧客は翌朝8:00にメールを受信（深夜送信なし）
```

**BAD例**:
```
現在時刻: 夜間モード（22:00）
→ 保留フラグなしで承認リクエスト送信
→ 担当者が23:00に承認
→ D3: 即時実行
→ 顧客が深夜23:05にフォローアップメールを受信
→ 顧客: 「深夜にメールを送ってくる会社は信頼できない」（夜間制約違反）
```

### エラー 3: コンテキストなし承認リクエストによる全件却下

**症状**: 承認リクエストに「フォローアップメール送信を承認してください」という1行のみが記載され、担当者がどの商談・何のメールかを判断できず全件却下する。
**原因**: Step 2のコンテキスト付きメッセージ生成を省略したシンプルな送信。
**対処**: 全承認リクエストに商談名・金額・ステージ・クロージング日・検出理由・下書き内容（send_emailの場合）を必ず含める。

**GOOD例**:
```
承認リクエスト:
  対象商談: 株式会社H — Enterpriseプラン（¥12M、Proposal、明日クロージング）
  推奨アクション: フォローアップメール送信
  理由: 3営業日未返信（4/1送信）
  下書き: 「田中様、提案書につきまして...」（全文表示）
→ 担当者が1件の情報で承認/却下を判断
→ 承認率が向上し、重要アクションが確実に実施
```

**BAD例**:
```
承認リクエスト: 「フォローアップメール送信を承認してください」
→ 担当者: 「何のメール？誰に？なぜ？」と困惑
→ 全件却下
→ 重要なクロージング前日のフォローアップが実施されず商談を逸失
```

### エラー 4: Slack障害時のフォールバック未実装

**症状**: Slack APIがダウン中に承認リクエストの送信が全て失敗し、承認制アクションが全て次サイクルに繰り延べられる。P1アクション（当日クロージング）が対応されないまま1時間が経過する。
**原因**: Step 5のメールフォールバックが未実装。
**対処**: Slack送信失敗時はメールフォールバックを自動的に試行する。両方失敗した場合のみ繰り延べとする。

**GOOD例**:
```
Slack API: 503 Service Unavailable
→ メールフォールバック: {担当者メールアドレス} へ送信
→ 送信成功
→ 承認リクエスト配信完了（チャンネルがSlackからメールに変わっただけ）
→ P1アクションの承認機会を失わない
```

**BAD例**:
```
Slack API: 503 Service Unavailable
→ メールフォールバック未実装
→ 承認リクエスト送信失敗 → 次サイクル繰り延べ
→ 当日クロージング商談（P1）が1時間以上対応されない
→ 担当者が気づかず商談期限切れ
```
