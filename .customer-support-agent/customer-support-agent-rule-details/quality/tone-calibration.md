# Q3: Tone Calibration（トーンキャリブレーション）

## Purpose

コンテンツレビュー済み出力ドラフトのトーンを、`common/tone-guidelines.md` のトーンマトリクス（セグメント × 感情）と照合し、不適合を自動修正する。クレーム対応の3ステップ構造（共感→謝罪→解決策）を検証する。トーンマトリクスが「即エスカレーション」を指示し、かつ現在のモードが即時回答の場合は R4（Escalation Handoff）へのリペアパスを発動する（最大1回）。

**Classification**: ALWAYS EXECUTE

**参照元**: `core-workflow.md` Q3、`common/tone-guidelines.md`

---

## Prerequisites

- Q2（Content Review）が完了し、コンテンツレビュー済みドラフトが渡されていること
- T1で検出された感情シグナル（`normal` / `confused` / `frustrated` / `angry`）が参照可能であること
- 顧客セグメント（`VIP` / `General` / `Complaint-in-progress`）が参照可能であること
- 現在の処理モード（`immediate` / `escalation` / `analysis`）が記録されていること
- リペアカウンター（初期値: 0）が参照可能であること

---

## Execution Steps

### Step 1: トーンマトリクス照合

T1の感情シグナルと顧客セグメントからトーンマトリクスの対象セルを特定する。

**トーンマトリクス（概要）**:

| セグメント＼感情 | normal | confused | frustrated | angry |
|--------------|--------|----------|------------|-------|
| VIP | 丁寧+迅速強調 | 丁寧+原因説明 | 丁寧+謝罪+迅速解決 | **即エスカレーション** |
| General | 標準丁寧語 | 手順ガイド強化 | 謝罪+解決策提示 | **即エスカレーション** |
| Complaint-in-progress | 共感+経緯確認 | 共感+原因説明 | 強化共感+謝罪 | **即エスカレーション** |

**「即エスカレーション」セル**: VIP×angry、General×angry、Complaint-in-progress×angry の3セル。  
これは `core-workflow.md` R1の感情ベースエスカレーション条件と同一の定義。

特定したセルのトーン要件をチェックリストとして展開する:

```
トーン要件チェックリスト（{セグメント} × {感情}）:
□ 要件1: {トーンマトリクス規定の要素1}
□ 要件2: {トーンマトリクス規定の要素2}
□ 要件3: （該当セルに応じて）
```

### Step 2: 3ステップ確認（クレーム対応セグメント）

顧客セグメントが `Complaint-in-progress` の場合、または感情シグナルが `frustrated` / `angry` の場合、ドラフトが以下の3ステップ構造を含むか確認する。

**3ステップ構造チェック**:
1. **共感パラグラフ**: 顧客の感情・状況への理解を示す文が冒頭に存在するか
2. **謝罪表現**: 適切な謝罪が含まれているか（過剰謝罪・責任転嫁は禁止）
3. **解決策提示**: 具体的な次のステップまたは解決策が明示されているか

いずれかが欠けている場合 → `STRUCTURE_INCOMPLETE` フラグを立て、自動補完を試みる。

**自動補完テンプレート（共感パラグラフ不足時）**:
```
{顧客名}様、この度はご不便とご心配をおかけしており、誠に申し訳ございません。
ご状況を詳しくお伺いしたうえで、早急に対応いたします。
```

**自動補完テンプレート（謝罪不足時）**:
```
本件につきまして、ご迷惑をおかけしたことを心よりお詫び申し上げます。
```

### Step 3: トーン不適合の自動修正

Step 1のチェックリストで未達の要件について、フレーズ置き換えと丁寧度調整を行う。

**修正方針**:

| 不適合タイプ | 修正内容 |
|------------|---------|
| 丁寧度不足（VIPへの普通語） | 丁寧語・尊敬語へ変換 |
| 迅速対応強調の欠落（VIP） | 「優先対応」「本日中にご連絡」等を追記 |
| 手順ガイドの不足（confused） | ステップ番号付きリストに再構成 |
| 共感表現の欠落（frustrated/Complaint） | 自動補完テンプレートを冒頭に挿入 |
| テンプレート的応対（Complaint×any） | パーソナライズ要素（顧客名・履歴参照）を追加 |

自動修正件数を `TONE_AUTO_FIXED: {N}件` としてログに記録する。

### Step 4: リペアパス判定

**「即エスカレーション」セルへの到達チェック**:

```
IF トーンマトリクスが「即エスカレーション」を指示
AND 現在モード == "immediate_response"
THEN
  リペアカウンターを確認:
    カウンター == 0:
      → リペアカウンターを 1 にインクリメント
      → R4（Escalation Handoff）へリペアパスを発動
      → R4出力は再び Q1 から品質検査ループへ入る
    カウンター >= 1:
      → 強制エスカレーション申告（2回目は Q4 で強制エスカレーション処理）
      → "FORCED_ESCALATION" フラグを立て Q4 へ渡す
ELSE
  → トーン修正済みドラフトを Q4 へ渡す
```

**リペアパスの動作**:
- R4 に渡す際、「Q3リペアパスによるエスカレーション」であることをコンテキストに明記
- R4 は即時回答ドラフトを破棄し、中間応答（「担当者に引き継ぎます」テンプレート）を新たに生成
- R4 の出力は Q1 → Q2 → Q3 → Q4 を再度通過する（ループ1回分）

### Step 5: チェック結果サマリー作成

```
[TONE CALIBRATION LOG]
- Timestamp: {ISO 8601}
- Ticket ID: {TKT-XXXXXXXX-NNN}
- Segment: {VIP | General | Complaint-in-progress}
- Emotion Signal: {normal | confused | frustrated | angry}
- Tone Matrix Cell: {セル名}
- Tone Requirements Met: {N}/{全要件数}
- TONE_AUTO_FIXED: {N}件
- Complaint 3-Step Check: {OK | STRUCTURE_INCOMPLETE → 補完済み}
- Repair Path: {none | ESCALATION_REPAIR_1 | FORCED_ESCALATION}
- Result: PASS | REPAIRED | FORCED_ESCALATION
```

## Question Generation

Q3は原則として自動処理。ただし以下の場合にオペレーターへ通知する。

**リペアパス発動時（1回目）**:
```
### トーン検査 — エスカレーション転換
感情シグナル「怒り」× セグメント「{セグメント}」により、
トーンマトリクスが即エスカレーションを指示しました。

現在: 即時回答モード → R4（エスカレーション）にリペアパスで切り替えます。
（Q3リペアパス: 1/1回目。2回目の場合は強制エスカレーション）

### WHAT'S NEXT
**A) エスカレーション内容を確認・修正する** — R4の引き継ぎ内容を調整
**B) エスカレーションを続行する** — R4 → Q1-Q4 パスで処理を続行
```

**強制エスカレーション時（2回目）**:
```
### トーン検査 — 強制エスカレーション
Q3リペアパスが2回目に到達しました。ループ制御により強制エスカレーションを実行します。
Q4にてエスカレーションモードで処理を完結させます。

### WHAT'S NEXT
**A) 強制エスカレーションの内容を確認する** — Q4通過前に内容を確認
**B) Q4（Dispatch）へ続行** — 強制エスカレーションのまま処理を完了
```

## Error Handling

### エラー1: 感情シグナルが記録されていない

**シナリオ**: T1で感情シグナルが未設定（フォールバック未発動）

**原因候補**:
- Pitfall 3（エスカレーション判定ミス）の前段: T1の感情検出ロジックの不具合
- Pitfall 4（トーンミスマッチ）のリスク: セグメントは判定できていても感情が不明

**対応**:
1. デフォルト感情シグナルを `normal` として処理を続行
2. `EMOTION_SIGNAL_MISSING` ログを記録（T1の問題として追跡）
3. セグメントが `Complaint-in-progress` の場合は、感情 `frustrated` として扱い（保守的判定）

### エラー2: 自動修正でトーン要件が満たせない

**シナリオ**: 回答がほぼ無内容（「確認します」の1行のみ）で、トーン修正の対象テキストが不足

**対応**:
1. R3（Draft Generation）に差し戻しを検討
2. SLAが許容する場合: R3再生成フラグを立て `DRAFT_INSUFFICIENT` を記録
3. SLA残時間 < 15分の場合: 最低限のテンプレート（共感+謝罪+「担当者より改めてご連絡します」）を用意してQ4へ渡す

### エラー3: リペアパス後の2回目Q3で再び「即エスカレーション」判定

**シナリオ**: R4で中間応答ドラフトを生成したが、再度Q3で「即エスカレーション」を判定

**対応**:
1. リペアカウンターが1の場合 → `FORCED_ESCALATION` フラグを立て Q4 に申告
2. Q4でエスカレーションモードとして処理を強制完結
3. ループ制御のカウンター確認を優先する（Pitfall 3の変形）

### エラー4: VIP顧客へのテンプレート的応対を見逃した場合

**シナリオ**: VIP顧客に対し、セグメントを `General` と誤判定したまま標準トーンで処理

**原因**: Pitfall 4（トーンミスマッチ）典型パターン

**対応**:
1. T1の顧客情報を再確認し、セグメントの修正を促す
2. 修正後にQ3を再実行（ただしリペアパスカウンターはセグメント誤判定修正では消費しない）
3. `SEGMENT_MISMATCH_DETECTED` ログを記録

## GOOD / BAD 具体例

### GOOD 例1: 怒りシグナル検出 → リペアパス発動

**状況**: セグメント `Complaint-in-progress`、感情 `angry`、現在モード `immediate_response`

**BAD（リペアパス未発動）**:
```
ご不明な点はサポートページをご確認ください。
（KB引用: https://kb.example.com/faq）
```
怒りシグナル × Complaint-in-progress のトーンマトリクスは「即エスカレーション」だが、即時回答モードのまま処理。Pitfall 4（トーンミスマッチ）直撃。

**GOOD（Q3リペアパス発動後 — R4処理）**:
```
【担当者より改めてご連絡します】
{顧客名}様、この度は重ねてご不便をおかけしており、誠に申し訳ございません。
お客様のご状況を専任の担当者がお伺いし、直接ご対応させていただきます。
本日{時間}までに改めてご連絡いたします。
```
トーンマトリクスに従いR4へ切り替え。中間応答テンプレートを使用してQ1-Q4を再通過。

### GOOD 例2: クレーム対応3ステップ補完

**BAD（3ステップ不完全）**:
```
ご質問の設定方法についてご案内します。
1. 設定画面を開く
2. 「通知設定」をクリック
3. ONに変更
```
Complaint-in-progress セグメントなのに共感パラグラフも謝罪もない。`STRUCTURE_INCOMPLETE`。

**GOOD（自動補完後）**:
```
{顧客名}様、この度はご不便とご心配をおかけしており、誠に申し訳ございません。
ご状況を詳しくお伺いしたうえで、早急に対応いたします。

本件につきまして、ご迷惑をおかけしたことを心よりお詫び申し上げます。

設定方法についてご案内いたします。
1. 設定画面を開く
2. 「通知設定」をクリック
3. ONに変更
```
共感パラグラフ・謝罪を冒頭に自動補完。3ステップ構造を充足。

### BAD 例1: VIP顧客へのテンプレート的応対を見逃し

**BAD（見逃し）**:
```
お問い合わせありがとうございます。
ご申告の件について確認いたします。
```
VIPセグメントに対して「迅速対応強調」「優先対応」の記述が一切ない。Pitfall 4（トーンミスマッチ）。

**GOOD（トーン修正後）**:
```
{顧客名}様、いつもご利用いただきありがとうございます。
本件は優先対応といたします。本日中にご回答できるよう進めてまいります。
```

### BAD 例2: 強制エスカレーション後の誤った処理継続

**BAD（NG行動）**:
Q3リペアパスが2回目に到達したにもかかわらず、カウンター確認を省略してさらにR4へ3回目のリペアを試みる。ループが暴走し処理が終了しない。

**GOOD（正しい処理）**:
リペアカウンターが1の状態で再度「即エスカレーション」判定 → `FORCED_ESCALATION` フラグを立て、Q4に強制エスカレーションとして渡す。ループ制御を最優先。
