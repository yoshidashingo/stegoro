# F1: Case Tracking（ケーストラッキング）

## Purpose

Q4（Dispatch）完了後、エスカレーション済み・保留回答送信後・顧客返信待ちのチケットを継続追跡する。チケット管理システムへの状態更新、SLA再計算、クローズ条件チェック、オペレーターへの解決確認を担う。即時回答で解決済みの案件はスキップし、追加連絡が発生した場合にのみ実行する。

**Classification**: CONDITIONAL EXECUTE

**参照元**: `core-workflow.md` F1、`common/session-continuity.md`

---

## Prerequisites

**Execute IF（いずれかに該当）**:
- Q4でエスカレーションモードが完結し、`Follow-up Required: YES` が記録されている
- R3の回答に「確認の上回答します」テンプレートが使用された（KB信頼度 0.4〜0.6 の保留回答）
- 顧客からの返信が来ており、元チケットにスレッドが紐づいている

**Skip IF**:
- 即時回答で完結し、Q4で `Follow-up Required: NO` が記録されている
- かつ顧客からの追加連絡が確認されていない

---

## Execution Steps

### Step 1: チケット状態の更新

チケット管理システムの状態を以下のルールで更新する。

**チケット状態定義**:

| 状態 | 遷移条件 |
|------|---------|
| `Open` | 新規受付（T1で作成） |
| `Pending` | 「確認の上回答します」送信後、または顧客返信待ち |
| `Escalated` | R4エスカレーション完了後（オペレーター引き継ぎ済み） |
| `Resolved` | 解決済み（オペレーターが解決報告 or 顧客確認済み） |
| `Closed` | クローズ条件充足後（後述） |

**チケット管理システム連携の責務定義（v2）**:

| 操作 | 実行フェーズ | 実行条件 |
|------|-----------|---------|
| `create` | T1 | 新規チケット受付時 |
| `update` | R1, Q4 | モード変更・送信完了時 |
| `escalate` | R4 | エスカレーション完了時 |
| `resolve` | F1 | クローズ条件充足時 |
| `close` | F1 | resolve後の最終クローズ |

### Step 2: 次回アクション日時の設定

エスカレーション後の追跡スケジュールを設定する。

```
次回アクション日時の設定ルール:
  エスカレーション（Escalated）:
    → オペレーター引き継ぎから 4時間 後を次回確認日時に設定
    → セキュリティインシデントは 1時間 後（プラン問わず）

  保留回答（Pending）:
    → 保留回答送信から 24時間 後を次回確認日時に設定
    → Enterpriseプランは 4時間 後

  顧客返信待ち:
    → 最終送信から 48時間 後を次回確認日時に設定
```

### Step 3: SLA再計算

エスカレーション後の解決SLAを再計算する。

**エスカレーション解決SLA**:

```
エスカレーション解決SLA（オペレーター引き継ぎ後）:
  デフォルト: 引き継ぎから 4時間
  セキュリティインシデント: 引き継ぎから 1時間
  Enterpriseプラン: 引き継ぎから 2時間
  P1（Critical）: 引き継ぎから 1時間（プラン問わず）

SLA再計算後の状態更新:
  チケット状態ファイルに追記:
  - Escalation SLA Target: {ISO 8601 datetime}
  - Escalation SLA Remaining: {分}分
```

**SLA超過時**（エスカレーション解決SLAが超過した場合）:
1. 管理者へ自動通知（Q4の管理者通知テンプレートと同形式）
2. オペレーターチームへのリマインダーを送信
3. 優先度をP1に昇格（既にP1の場合は管理者へエスカレーション）

### Step 4: クローズ条件チェック

以下のいずれかを充足した場合、チケットを `Resolved` → `Closed` に遷移させる。

```
クローズ条件:
  条件A: 顧客確認済み
    → 顧客から「解決しました」「ありがとうございました」等の確認メールを受信
    → F2（FAQ候補抽出）トリガー判定を行ってから Close

  条件B: 7日間返信なし
    → 最終送信から7日間（168時間）顧客からの返信がない
    → 自動クローズの前に1回だけ確認メールを送信:
       「解決されましたでしょうか。ご不明な点があればいつでもご連絡ください」
    → 確認メールから48時間返信なし → Closed

  条件C: オペレーター解決報告
    → 担当オペレーターがチケット管理システムで「解決済み」を報告
    → 顧客へ解決通知メールを送信してから Closed
```

**クローズ時のCSAT付与**（エスカレーション案件）:
```
エスカレーション解決後のCSATアンケート:
---
この度のサポートはいかがでしたでしょうか？
担当者との対応についてお聞かせください。
[満足度を回答する（1〜5）]: {CSAT_SURVEY_URL}
---
```

### Step 5: フォローアップ監査ログ記録

```
[F1 CASE TRACKING LOG]
- Timestamp: {ISO 8601}
- Ticket ID: {TKT-XXXXXXXX-NNN}
- Trigger: {escalation | pending_response | customer_reply}
- Previous Status: {Open | Pending | Escalated}
- New Status: {Pending | Escalated | Resolved | Closed}
- SLA Recalculated: {YES / NO}
  - New SLA Target: {ISO 8601 datetime}
  - SLA Status: {OK | Warning | Exceeded}
- Next Action: {datetime} / {action type}
- Close Condition Met: {YES: 条件A/B/C | NO}
- CSAT Sent: {YES | NO | PENDING}
- F2 Trigger Check: {YES（閾値超過） | NO}
```

## Output Artifacts

- **チケット状態更新**: チケット管理システムへの状態遷移記録
- **次回アクション日時**: 追跡スケジュール
- **SLA再計算結果**: エスカレーション解決SLA
- **クローズ処理**: Resolved → Closed 遷移（条件充足時）
- **CSATアンケート**: クローズ時に付与（エスカレーション案件）
- **F2トリガーフラグ**: FAQ候補抽出の実行判定
- **フォローアップ監査ログ**: 監査ログに追記

## Error Handling

### エラー1: チケット管理システムへの接続失敗

**シナリオ**: チケット管理システムAPIがエラーを返し、状態更新ができない

**対応**:
1. 3回まで自動リトライ
2. 失敗した場合、チケット状態をローカルのセッション状態ファイルに記録（`common/session-continuity.md` 参照）
3. `TICKETING_SYSTEM_ERROR` ログを管理者に通知
4. 次回アクション日時を手動フォローアップリストに追加

### エラー2: エスカレーション後にオペレーターから応答がない

**シナリオ**: エスカレーション解決SLA（4時間）を超過してもオペレーターから解決報告がない

**原因候補**:
- Pitfall 3（エスカレーション判定ミス）: 不適切なチームにエスカレーションされた可能性
- オペレーターの不在・過負荷

**対応**:
1. オペレーターチームにリマインダーを送信
2. 管理者に SLA超過通知を送信
3. P1以外の案件は優先度を1段階昇格（P3→P2等）
4. 12時間以上応答がない場合、別の担当チームへの再エスカレーションを管理者に申告

### エラー3: 顧客からの返信がエスカレーション中に重複チケットを生成した場合

**シナリオ**: 顧客が返信メールを別件として送信し、T1で新規チケットが作成された

**対応**:
1. T1の重複検出ロジック（24時間以内の同一顧客）でキャッチされているか確認
2. キャッチされていない場合、既存の `Escalated` チケットに手動で紐づけ
3. 新規チケットは `Duplicate` ステータスでクローズ
4. 既存チケットの反復カウンターをインクリメント

### エラー4: クローズ条件B（7日返信なし）で顧客がその後連絡してきた

**シナリオ**: 7日間未返信でクローズ処理を行った後、顧客から問題が解決していないと連絡

**対応**:
1. クローズされたチケットを `Reopened` 状態に戻す
2. 元の対応経緯を引き継いだ新しいチケットとして処理（反復カウンター引き継ぎ）
3. 対応経緯のある顧客へは Complaint-in-progress セグメントに昇格を検討
4. `TICKET_REOPENED` ログを記録

## GOOD / BAD 具体例

### GOOD 例1: エスカレーション後48時間での解決確認

**状況**: Enterpriseプランのセキュリティ案件をエスカレーション。解決SLA: 2時間。

**GOOD（F1処理フロー）**:
1. Q4完了直後: チケット状態を `Escalated` に更新。エスカレーション解決SLA: 引き継ぎから2時間
2. 2時間後: SLA確認アクション発火。オペレーターから解決報告あり
3. 顧客へ解決通知メール送信: 「担当者より解決のご連絡を申し上げます」
4. CSATアンケート付与してから `Resolved` → `Closed`
5. 解決時間48分でクローズ完了をログ記録

**解説**: SLA達成での解決。クローズ条件C（オペレーター報告）が充足。

### GOOD 例2: 保留回答（確認の上回答）後のフォローアップ

**状況**: KB信頼度 0.52 で「確認の上回答します」を送信。チケット状態 `Pending`。

**GOOD（F1処理フロー）**:
1. 次回アクション: 24時間後に確認
2. 24時間後: 担当チームからの確認回答を受信 → 新規回答ドラフトを生成（R3から再処理）
3. 新回答が Q1-Q4 を通過 → 顧客へ送信
4. チケット `Resolved` に更新

### BAD 例1: エスカレーション後放置で顧客から再問い合わせ

**BAD（NG行動）**:
R4でエスカレーション完了後、F1をスキップ。4時間後のアクションを設定せず追跡しない。

**結果**: オペレーターが対応を忘れ、顧客から「まだ解決していない」という再問い合わせ。Pitfall 3（エスカレーション判定ミス）の事後発生。

**GOOD**: Q4完了後にF1を即座に実行し、エスカレーション解決SLAを設定・監視。

### BAD 例2: クローズ条件を誤適用してアクティブ案件をクローズ

**BAD（NG行動）**:
顧客から48時間返信がないことを「条件B（7日返信なし）充足」と誤判断し、4日目にクローズ。

**GOOD（正しい処理）**:
条件B は「最終送信から7日間（168時間）」を厳密に適用。確認メール送信後さらに48時間待つ。合計最短9日間が条件B適用の最短期間。
