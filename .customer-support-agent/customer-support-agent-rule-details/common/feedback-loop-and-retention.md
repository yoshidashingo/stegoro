# Feedback Loop and Retention — CSATフィードバックループ・データ保持ポリシー

## このファイルの目的

CSATフィードバックを品質改善に還元するループと、監査ログ・チケットデータの保持期間・アクセス制御・削除ポリシーを定義する。
`core-workflow.md` Q4（CSAT自動添付）および F2（FAQ候補抽出）との連携仕様を含む。
**v2新規追加**（Common Rules Design M5 fix 対応）。

## CSATフィードバックループ

### CSAT収集

- Q4ディスパッチ後、即時回答・エスカレーション（顧客向け暫定回答）の両モードでCSATサーベイを自動添付する（設定で無効化可能）。
- 評価スコア: 1〜5の5段階（1: 非常に不満 / 5: 非常に満足）
- 任意コメント欄: 自由記述（PIIを含む可能性があるため収集後にPIIスキャンを実施）

### CSAT低評価の自動フラグ

| 条件 | アクション |
|------|----------|
| スコア <= 3 | チケットに `csat_low_flag` を設定、週次レビューキューに追加 |
| スコア <= 2 | 即時マネージャー通知 |
| スコア = 1 | 即時マネージャー通知 + 当該オペレーター/エージェントへの通知 |

### 週次レビュープロセス

毎週月曜日（営業時間開始時）に自動実行:

```
Step 1: csat_low_flag が立ったチケットを集計（前週分）
Step 2: カテゴリ別・セグメント別・感情シグナル別に分類
Step 3: パターン分析（同一カテゴリでの反復低評価、特定オペレーターへの集中等）
Step 4: 改善アクション候補を生成（以下の3種類）
  a. KB記事更新候補（回答の根拠となったKB記事の改善）
  b. トーンテンプレート修正候補（トーン関連の低評価）
  c. エスカレーション閾値調整候補（過剰/過少エスカレーションの改善）
Step 5: 改善アクション候補をマネージャーへ提出（CP-承認待ち）
```

## FAQ候補抽出との連携（F2）

F2 ステージのトリガー条件:

| トリガー | 閾値 |
|---------|------|
| 同一カテゴリ × 同一サブカテゴリの問い合わせ件数 | >= 5件/週 |
| CSAT スコア <= 3 の蓄積件数 | >= 3件（同一カテゴリ） |

トリガー発動時の処理:

```
1. 重複問い合わせをクラスタリング（カテゴリ × サブカテゴリ × キーワード）
2. KB未整備領域の特定（該当カテゴリのKBカバレッジ確認）
3. CSAT低スコアとの相関分析
4. KB改善提案を生成（F2 KB改善提案テンプレート使用）
```

### KB改善提案テンプレート

```
【KB改善提案】
問題サマリー   : {反復している問い合わせの概要}
推奨KB内容    : {追加・更新すべきKBの内容}
影響チケット数 : {該当期間のチケット件数}
CSAT平均     : {該当チケットのCSAT平均スコア}
優先度       : High / Medium / Low
提案日       : {ISO 8601 date}
```

## 改善アクションの実施サイクル

```
CSAT収集（Q4）
    ↓
低評価フラグ（自動）
    ↓
週次レビュー（自動集計）
    ↓
改善アクション候補生成（自動）
    ↓
マネージャー承認（人間レビュー）
    ↓
改善実施（KB更新 / テンプレート修正 / 閾値調整）
    ↓
次週のCSATで効果測定
```

## 監査ログ保持期間・アクセス制御

### 保持期間テーブル

| データ種別 | 保持期間 | アクセス権限 | 保存形式 |
|-----------|---------|------------|---------|
| チケットメタデータ（ID・カテゴリ・優先度・SLAステータス） | 3年 | サポートチーム全員 | 構造化ログ |
| 対応履歴（回答内容・エスカレーションサマリー・マスキング済み） | 1年 | 担当者 + マネージャー | 構造化ログ |
| PIIを含む生データ（顧客メール原文・マスキング前データ） | 90日 | マネージャーのみ | 暗号化ストレージ |
| 品質チェック結果（Q1〜Q4の全チェック結果） | 3年 | コンプライアンスチーム | 監査ログ |
| CSATスコア・フィードバックコメント（PIIマスキング後） | 1年 | サポートチーム全員 | 構造化ログ |
| PIIを含むCSATコメント原文 | 90日 | マネージャーのみ | 暗号化ストレージ |
| 監査ログ（エラー・リペアパス・SLA超過記録） | 3年 | コンプライアンスチーム | 監査ログ |

### アクセス制御の原則

- **最小権限の原則**: 業務上必要な最小限のデータにのみアクセスできる
- **PIIデータへのアクセスログ**: マネージャーがPIIデータにアクセスした場合、アクセスログを記録する
- **役割ベースのアクセス制御（RBAC）**:

| 役割 | アクセス可能データ |
|------|----------------|
| サポートエージェント | チケットメタデータ・自担当チケットの対応履歴・CSAT集計 |
| シニアサポート | 全チケットメタデータ・全対応履歴 |
| マネージャー | 上記すべて + PII生データ（業務上必要な場合） |
| コンプライアンスチーム | 品質チェック結果・監査ログ（PIIマスキング後） |

## 削除ポリシー

### 自動パージ

保持期間経過後は自動パージを実行する。

```
パージスケジュール: 毎日 02:00 JST（低トラフィック時間帯）
対象: 保持期間を超過したデータ
確認: パージ前に保持期間を再確認し、誤削除を防止
ログ: パージ実行ログを別途 180日間保持
```

### 顧客からの削除要求（GDPR準拠）

保持期間内であっても、顧客の削除要求に応じる。

```
Step 1: 顧客からの削除要求を受信（メール or 問い合わせフォーム）
Step 2: 本人確認を実施
Step 3: 対象データの特定（チケットID・期間で検索）
Step 4: PIIを含むデータを削除（対応履歴のPII部分をマスキングで代替可）
Step 5: チケットメタデータは匿名化して保持（統計目的）
Step 6: 削除完了を顧客へ通知（30日以内）
Step 7: 削除実行ログを記録
```

**注意**: 削除要求対応中のチケットに関するデータは、法的紛争リスクがある場合にのみ保持を延長できる（法務確認必須）。

## GOOD / BAD 例

### GOOD: CSATフィードバックループの活用（正しい）

```
週次レビューで「課金カテゴリのCSAT平均 2.8」を検出。
対応履歴を分析したところ、KB記事「プラン変更手順」が 95日前更新で古く、
手順が現在のUIと異なっていた。
KB改善提案を作成し、マネージャー承認後にKBを更新した。
翌週の課金カテゴリCSATは 3.6 に改善した。
```

### BAD: CSATフィードバックの放置（誤り）

```
CSAT低評価チケットが溜まっているが、週次レビューを実施せずに放置した。
```
理由: フィードバックループが機能せず、同じ問題が繰り返されKBが改善されない。品質劣化が加速する。

---

### GOOD: 削除要求への対応（正しい）

```
顧客から「自分のデータを削除してほしい」という要求を受信。
本人確認後、PIIを含む生データ（90日保持データ）を削除し、
チケットメタデータは匿名化（顧客IDを削除）して統計用に保持した。
30日以内に削除完了を顧客へ通知した。
```

### BAD: 削除要求の無視（誤り）

```
「削除要求はシステム上対応していないため、保持期間まで削除できません」と回答した。
```
理由: GDPR・個人情報保護法に基づく削除権（忘れられる権利）への対応義務を果たしていない。法的リスク。
