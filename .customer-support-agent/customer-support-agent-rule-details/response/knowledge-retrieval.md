# R2: Knowledge Retrieval — Phase Rule

## Purpose

ナレッジベース（KB）を検索し、問い合わせに関連する記事を取得する。取得結果の信頼度スコアを算出し、閾値に基づいて即時回答継続・確認テンプレート使用・エスカレーション切り替えの3ルートに分岐する。ハルシネーション防止のため、KB 外情報の生成を明示的に禁止する。

**Classification**: ALWAYS EXECUTE（即時回答モード時）
**Approval Gate**: なし（信頼度スコアにより自動分岐）
**参照元**: `core-workflow.md` R2 セクション

---

## Prerequisites

- R1 でモード選択が「即時回答モード」に確定していること
- `common/overconfidence-prevention.md` を読み込み済み（ハルシネーション防止ルールの参照元）
- KB システムへの検索権限があること

---

## Execution Steps

### Step 1: 検索クエリ構築

T2 の分類結果とメール本文から検索クエリを構築する。

**クエリ構築ロジック**:

1. **プライマリキーワード**: T2 の `matched_keywords` リストから上位 3 件を抽出
2. **セカンダリキーワード**: メール件名から名詞句を抽出（形態素解析）
3. **カテゴリフィルター**: T2 の `primary_category` を検索スコープの絞り込みに使用
4. **顧客プランフィルター**: T1 の `plan` を KB 記事の適用対象フィールドにマッピング（例: Enterprise 向け記事を優先）

**クエリテンプレート**:
```
{プライマリキーワード} {セカンダリキーワード}
filter: category={primary_category}, applicable_plan={plan}
```

### Step 2: 3段階検索戦略

以下の順序で検索を実行し、各段階で信頼度が閾値（0.6）に達した場合は後続段階をスキップする。

#### フェーズ 1: キーワード検索

- 検索対象: KB 記事のタイトル・タグ・キーワードフィールド
- 完全一致 > 部分一致の順でスコアリング
- 結果: 最大 10 件取得

#### フェーズ 2: セマンティック検索

- フェーズ 1 の結果が不十分（上位3件の信頼度 < 0.6）な場合に実行
- 検索対象: KB 記事の本文全体を意味的類似度でランキング
- ベクトル類似度スコアをフェーズ 1 のスコアと統合
- 結果: 最大 5 件取得（フェーズ 1 結果と統合してランク付け）

#### フェーズ 3: カテゴリフィルタリング

- フェーズ 1・2 の結果が信頼度閾値に達しない場合、カテゴリ絞り込みを緩和して再検索
- `primary_category` フィルターを外し、全カテゴリで検索
- 過去の類似チケット解決事例も検索対象に含める

### Step 3: 検索結果ランキング

各 KB 記事に対して以下の3軸でスコアを算出し、加重平均で最終スコアを計算する。

| 軸 | 重み | 計算方法 |
|---|-----|---------|
| 関連度（Relevance） | 0.5 | クエリとのキーワード一致度・意味的類似度 |
| 鮮度（Freshness） | 0.3 | 最終更新日からの経過日数（30日以内: 1.0、90日以内: 0.7、90日超: 0.4） |
| 信頼度（Confidence） | 0.2 | 過去の参照回数・解決率に基づく記事品質スコア |

**最終スコア算出**:
```
最終スコア = (関連度 × 0.5) + (鮮度 × 0.3) + (信頼度 × 0.2)
```

**KB 記事鮮度チェック（v2）**:
- 最終更新日が 90 日以上前の記事: `freshness_warning: true` フラグを付与
- Q2（Content Review）でこのフラグを参照し、内容の正確性を強化チェックする

### Step 4: KB 信頼度スコア閾値判定（v2）

最上位 KB 記事の最終スコアに基づき、以下の3ルートに分岐する。

```
最終スコア >= 0.6
  → 即時回答モード継続 → R3（Draft Generation）へ
  → retrieved_articles: スコア上位3件をR3に引き渡す

最終スコア 0.4〜0.6（未満）
  → 「確認の上回答」テンプレート使用モード → R3（Draft Generation）へ
  → r3_mode: "will_confirm" をチケットに記録
  → retrieved_articles: スコア上位3件をR3に引き渡す（参考情報として）

最終スコア < 0.4 または KB 記事 0 件
  → エスカレーションモードに切り替え → R4（Escalation Handoff）へ
  → escalation_reason: "low_kb_confidence"
  → retrieved_articles: 空（KB 情報なし）
```

**重要**: スコア 0.4〜0.6 のゾーンでは「確認の上回答」テンプレートを使用するが、KB 外情報を補完生成してはならない。参考として引き渡す KB 記事は「参考情報」として明示し、断定的な回答に使用しない。

### Step 5: 過去類似チケット解決事例の参照

反復問い合わせ（`repeat_count >= 1`）の場合、過去の対応経緯を統合する。

**参照対象**:
- 同一顧客の同カテゴリ・サブカテゴリのクローズ済みチケット
- 解決方法（resolution summary）と適用した KB 記事 URL
- 解決にかかった往復回数と CSAT スコア（利用可能な場合）

**統合方法**: 過去解決事例を KB 検索結果と並列に保持し、R3 へ渡す。R3 はドラフト生成時に「前回の対応」として参照する。

### Step 6: 結果のチケットへの記録

```yaml
knowledge_retrieval:
  search_phases_executed: "{[keyword] | [keyword, semantic] | [keyword, semantic, category_relaxed]}"
  top_kb_score: "{0.0 - 1.0}"
  kb_route: "{immediate | will_confirm | escalation}"
  retrieved_articles:
    - article_id: "{KB記事ID}"
      title: "{記事タイトル}"
      url: "{記事URL}"
      score: "{最終スコア}"
      freshness_warning: "{true | false}"
  past_resolution_count: "{参照した過去解決事例の件数}"

audit_log:
  - timestamp: "{ISO 8601}"
    stage: "R2"
    action: "kb_searched"
    details: "top_score: {score}, route: {kb_route}, articles: {件数}"
```

## Question Generation

R2 は Approval Gate なし。信頼度スコアに基づく自動ルーティングのみ。

---

## Output Artifacts

R2 が生成するアーティファクト:

1. **KB 検索結果**（チケットオブジェクトの `knowledge_retrieval` フィールドに追記）— R3/R4 で参照
2. **KB ルート判定**（`kb_route`）— R3 のドラフト生成モードを制御
3. **監査ログエントリ** — `stage: R2, action: kb_searched`

## Error Handling

### エラー 1: 低信頼度（0.2）でも「確認の上回答」テンプレートを使用（Pitfall: ハルシネーション回答）

**症状**: スコア 0.2 の KB 記事を参考に「確認の上回答します」テンプレートを生成するが、参考記事の情報が誤誘導となりR3 で推測回答が混入する。  
**原因**: 閾値の境界値判定が不正確（`< 0.4` を `< 0.3` と実装するなど）。  
**対処**: スコア < 0.4 は例外なくエスカレーション（R4）へルーティングする。0.4〜0.6 ゾーンの「確認の上回答」テンプレートでも、参考記事情報を断定的な回答文に使用しない。

**GOOD例**:
```
最高 KB スコア: 0.30
→ スコア < 0.4 → エスカレーションモードに切り替え
→ escalation_reason: "low_kb_confidence"
→ R4 で「専門担当者に確認の上、改めてご連絡します」テンプレートを作成
→ Q1-Q4 を通過して送信
```

**BAD例**:
```
最高 KB スコア: 0.30
→ 実装誤り: 0.30 < 0.4 の判定が 0.30 < 0.3 と実装されており「確認の上回答」に分岐
→ R3 でスコア 0.30 の KB 記事を参考に手順を生成
→ 存在しない機能への手順を案内（ハルシネーション）
→ 顧客がその手順を試みて混乱
```

### エラー 2: KB 外情報の補完生成（Pitfall: ハルシネーション回答）

**症状**: KB 検索結果が不十分なため、LLM が「それらしい」手順や機能名を生成して回答に含める。  
**原因**: `common/overconfidence-prevention.md` のハルシネーション防止ルールが適用されていない。  
**対処**: R2 の出力として渡される `retrieved_articles` にない情報は、R3 でいかなる理由でも生成しない。R2 の段階で `retrieved_articles` が空の場合は必ず R4 へルーティングする。

**GOOD例**:
```
KB 検索結果: 0件（スコアなし）
→ R2: kb_route = "escalation"
→ R4 で引き継ぎサマリーを生成
→ 「KB に該当情報なし」を引き継ぎサマリーに明記
→ 人間オペレーターが調査して回答
```

**BAD例**:
```
KB 検索結果: 0件
→ R3 で「おそらくこういう手順だろう」と推測回答を生成
→「設定画面から〇〇オプションを有効にしてください」（存在しない機能）
→ 顧客が操作を試みるが機能が見つからず、追加問い合わせが発生
```

### エラー 3: 鮮度警告付き記事をそのまま利用しQA で問題発覚（Pitfall: ハルシネーション回答）

**症状**: 120 日前に更新された KB 記事（廃止済み機能への手順）を信頼度 0.7 として取得し、R3 でドラフトを生成。Q2 で廃止済み機能への言及として検出される。  
**原因**: `freshness_warning` フラグを R3/Q2 に渡していない。  
**対処**: `freshness_warning: true` の記事を使用する場合、R3 にその情報を引き渡し、Q2（Content Review）で強化チェック対象として処理する。可能であれば最新版記事への差し替えを試みる。

**GOOD例**:
```
KB 記事: 最終更新 100日前 → freshness_warning: true
→ スコア: 0.75（関連度は高い）
→ R2: retrieved_articles に freshness_warning: true で記録
→ R3: 鮮度警告記事を使用する旨をドラフトのメタデータに付記
→ Q2: 廃止済み機能・変更された手順がないか強化チェック
→ 問題発見 → 自動修正 or 人間レビューフラグ
```

**BAD例**:
```
KB 記事: 最終更新 100日前 → freshness_warning が記録されない
→ スコア 0.75 の記事をそのまま R3 に引き渡し
→ Q2 で通常チェックのみ → 廃止済み機能への手順がスルー
→ 顧客に廃止済み機能の操作手順を案内
```

### エラー 4: 反復問い合わせで過去解決事例を無視（Pitfall: エスカレーション判定ミス）

**症状**: `repeat_count: 2` のチケットで、同じ KB 記事から同じ回答を生成。前回の回答が無効だったのに同じ内容を再送。  
**原因**: Step 5 の過去類似チケット解決事例参照が未実装。  
**対処**: `repeat_count >= 1` の場合は必ず過去解決事例を検索し、前回の KB 記事・解決方法・CSAT スコアを R3 に渡す。前回と同一の KB 記事のみが取得された場合は `escalation_reason: "repeated_same_kb"` でエスカレーションを検討する。
