# R4: Escalation Handoff — Phase Rule

## Purpose

エスカレーション条件が成立した問い合わせを、人間オペレーターへ引き継ぐ。引き継ぎサマリーと顧客向け中間応答ドラフトを生成し、両者を Q1〜Q4 の品質チェックに通過させてから送信する。エスカレーション後の追跡を F1（Case Tracking）に引き渡す。

**Classification**: CONDITIONAL EXECUTE
**Execute IF**: R1 でエスカレーション判定 / R2 で KB 信頼度 < 0.4 / Q3 のリペアパス（感情トーン再判定）
**Skip IF**: 即時回答モード継続 and 分析モード
**Approval Gate**: あり（CP-3: エスカレーション内容の確認）
**参照元**: `core-workflow.md` R4 セクション

---

## Prerequisites

- R1 でエスカレーションモードが確定していること、または R2 で `kb_route: "escalation"` が確定していること、または Q3 からリペアパスで遷移していること
- `escalation_reason` がチケットオブジェクトに記録されていること
- `common/output-structure-patterns.md` を読み込み済み（引き継ぎサマリー・中間応答テンプレートの参照元）
- `common/tone-guidelines.md` を読み込み済み（中間応答のトーン設定）

---

## Execution Steps

### Step 1: エスカレーション先の選定

`primary_category` と `escalation_reason` に基づき、担当チームを選定する。

| カテゴリ / 理由 | エスカレーション先チーム |
|-------------|-------------------|
| Security（任意） | セキュリティ対応チーム（最優先） |
| `low_kb_confidence` | テクニカルサポート上位チーム |
| `data_loss_risk` | テクニカルサポート上位チーム + SRE チーム |
| `legal_issue` | 法務チーム + カスタマーサポート管理職 |
| `explicit_operator_request` | カスタマーサポート オペレーター（一般） |
| `emotion_based_angry_complaint` | クレーム専任チーム or シニアオペレーター |
| `repeated_unresolved` (3往復以上) | シニアオペレーター |
| Bug / Performance | テクニカルサポート |
| Billing | 課金サポートチーム |
| Feature | プロダクトサポートチーム |
| Account | アカウントサポートチーム |

### Step 2: 引き継ぎサマリーの生成

以下のテンプレートに従い、引き継ぎサマリーを生成する。

```
==== エスカレーション引き継ぎサマリー ====

【チケット情報】
- Ticket ID: {TKT-xxx}
- 受信日時: {ISO 8601}
- エスカレーション先: {担当チーム名}
- エスカレーション理由: {escalation_reason の日本語説明}

【顧客情報】
- 顧客メールアドレス: {マスキング済み — Q1 通過後に最終版が確定}
- セグメント: {VIP | General | Complaint-in-progress}
- プラン: {Free | Pro | Enterprise}
- 反復回数（repeat_count）: {整数}

【問い合わせ概要】
{メール本文の要点（最大200文字、PII は [REDACTED] でマスク）}

【カテゴリ・優先度】
- カテゴリ: {primary_category} / {subcategory}
- 優先度: {P1 | P2 | P3 | P4}
- SLA 期限: {ISO 8601} / ステータス: {on_track | warning | exceeded}

【対応経緯（試行済みアクション）】
{以下を箇条書きで記録}
- T1〜T3: トリアージ完了
- R1: {エスカレーション条件の詳細}
- R2（該当する場合）: KB 検索実施 / 最高スコア: {score} / 結果: {件数} 件
- Q3 リペアパス（該当する場合）: トーン再判定によりエスカレーション

【推奨アクション】
{escalation_reason に応じた推奨対応を記載}
例:
- Security: 「アカウントの一時停止と不正アクセスログの確認を優先してください」
- data_loss_risk: 「バックアップの確認と復旧可否の調査を行ってください」
- emotion_based: 「クレーム専任担当者が直接電話対応を検討してください」
- low_kb_confidence: 「KB に未記載の製品仕様について直接回答してください」

【感情シグナル】
- 検出レベル: {angry | frustrated | confused | normal}
- 対応上の注意: {感情レベルに応じた対応注意事項}

==== サマリー終了 ====
```

### Step 3: 中間応答ドラフトの生成（v2 必須）

顧客向けに「担当者へ引き継ぎ中」であることを伝える中間応答ドラフトを作成する。

**重要**: このドラフトは Q1〜Q4 を通過してから送信する。PII を含む状態で送信してはならない。

**中間応答テンプレート（escalation_reason 別）**:

#### 条件ベースエスカレーション（Security / legal / data_loss）

```
件名: Re: {元件名}

{顧客名} 様

この度はお問い合わせいただき、誠にありがとうございます。

ご連絡いただいた内容につきましては、専門の担当チームが対応させていただくため、
引き継ぎの手続きをとらせていただきました。

担当者より改めてご連絡申し上げますので、いましばらくお待ちください。
対応期限: {SLA期限に基づく目安（例: 本日中、翌営業日中）}

ご不便をおかけし、大変申し訳ございません。
引き続きどうぞよろしくお願いいたします。

{サービス名} カスタマーサポート
```

#### 感情ベースエスカレーション（angry × Complaint-in-progress）

```
件名: Re: {元件名}

{顧客名} 様

この度は大変ご不便・ご不満をおかけしており、誠に申し訳ございません。

お客様のご状況を真摯に受け止め、専任の担当者が直接ご対応させていただきます。
現在、担当者への引き継ぎ手続きを行っております。

折り返しのご連絡まで、今しばらくお時間をいただけますようお願い申し上げます。
目安: {SLA期限に基づく目安}

重ねてご迷惑をおかけしたことをお詫び申し上げます。

{サービス名} カスタマーサポート
```

#### KB 信頼度不足によるエスカレーション（low_kb_confidence）

```
件名: Re: {元件名}

{顧客名} 様

お問い合わせいただき、誠にありがとうございます。

ご質問の内容につきましては、担当部門に確認の上、正確な情報をご回答するため、
専門チームへ引き継ぎの対応をとらせていただきました。

調査の上、改めてご連絡させていただきます。
対応期限: {SLA期限に基づく目安}

お待たせしてしまい、大変申し訳ございません。

{サービス名} カスタマーサポート
```

### Step 4: Q3 リペアパス経由時の追加処理

Q3 からリペアパスで R4 に遷移した場合、以下の追加情報を引き継ぎサマリーに記録する。

```
【Q3 リペアパス情報】
- リペアパス発動回数: {1}（最大1回。2回目は強制エスカレーション）
- Q3 判定理由: トーンマトリクスで「即エスカレーション」セルに合致
- 元のドラフト: 即時回答モードで生成されたが、トーン再判定でエスカレーション必要と判断
```

### Step 5: F1（Case Tracking）への引き継ぎ

エスカレーション完了後、F1（Case Tracking）でのフォローアップを設定する。

- チケットステータスを `Escalated` に更新する
- 次回アクション日時: SLA 期限（エスカレーション後解決 SLA: デフォルト 4h。Security インシデントは別途）
- F1 に引き継ぎ先チーム名と引き継ぎ時刻を記録する

### Step 6: チケットへの記録

```yaml
escalation_handoff:
  escalation_team: "{担当チーム名}"
  escalation_reason: "{理由コード}"
  repair_path_triggered: "{true | false}"
  handoff_summary_generated: "{true | false}"
  interim_response_template: "{条件別テンプレート種別}"

audit_log:
  - timestamp: "{ISO 8601}"
    stage: "R4"
    action: "escalation_handoff_generated"
    details: "team: {team}, reason: {reason}, repair_path: {true|false}"
```

## Question Generation（CP-3）

R4 完了後、引き継ぎ内容を提示しユーザー確認を求める（CP-3）。

**CP-3 提示テンプレート**:

```
### REVIEW REQUIRED
エスカレーション引き継ぎの内容を確認してください。

**エスカレーション情報**
- 引き継ぎ先: {担当チーム名}
- エスカレーション理由: {escalation_reason の日本語説明}
- Q3 リペアパス: {あり | なし}

**引き継ぎサマリー（プレビュー）**
{引き継ぎサマリーの冒頭200文字}...

**中間応答ドラフト（プレビュー）**
{中間応答ドラフト全文}

※ この後、中間応答と引き継ぎサマリーはともに Q1〜Q4 の品質チェックを通過します。

### WHAT'S NEXT
**A) 修正依頼** — 引き継ぎ内容を修正する
**B) Q1（PII Scan）へ続行** — 品質チェックへ進む
```

---

## Output Artifacts

R4 が生成するアーティファクト:

1. **引き継ぎサマリー**（オペレーターチーム向け）— Q1〜Q4 通過後に担当チームへ送付
2. **中間応答ドラフト**（顧客向け）— Q1〜Q4 通過後に顧客へ送信
3. **F1 Case Tracking への引き継ぎ指示**
4. **監査ログエントリ** — `stage: R4, action: escalation_handoff_generated`

## Error Handling

### エラー 1: 中間応答に PII を含んだまま送信（Pitfall: PII 漏洩）

**症状**: 中間応答ドラフトに顧客のメールアドレスや電話番号が含まれているが、Q1 をスキップして送信。  
**原因**: 「エスカレーションは自動処理なので品質チェック不要」という誤った判断。  
**対処**: R4 で生成した中間応答・引き継ぎサマリーは、必ず Q1〜Q4 を通過する（core-workflow.md の MANDATORY ルール）。Q1 PII Scan はいかなる状況でも省略不可。

**GOOD例**:
```
中間応答ドラフト: 「田中様、〇〇-〇〇〇〇-〇〇〇〇にてご連絡いただいた件について...」
→ Q1 PII Scan: 電話番号パターン検出 → 「****-****-****」にマスキング
→ Q2〜Q4 を通過
→ マスキング済み中間応答を顧客へ送信
```

**BAD例**:
```
中間応答ドラフト: 「田中様、090-1234-5678 にてご連絡いただいた件について...」
→ Q1 をスキップ（「エスカレーションだから急ぐ」という判断）
→ 顧客の電話番号が回答メールに含まれた状態で送信
→ PII 漏洩インシデント（法的リスク）
```

### エラー 2: 完全なコンテキストなしの引き継ぎ（Pitfall: エスカレーション判定ミス）

**症状**: 引き継ぎサマリーに「問い合わせ内容を確認してください」とだけ記載され、エスカレーション理由・対応経緯・推奨アクションが欠如。人間オペレーターが最初から調査し直す羽目になる。  
**原因**: Step 2 の引き継ぎサマリーテンプレートを使用していない。  
**対処**: 引き継ぎサマリーは Step 2 のテンプレートに定義された全セクションを必ず含める。特に「試行済みアクション」と「推奨アクション」は漏れなく記載する。

**GOOD例**:
```
引き継ぎサマリー:
- エスカレーション理由: KB 信頼度スコア 0.25（閾値 0.4 未満）
- 対応経緯: キーワード検索・セマンティック検索実施、最高スコア 0.25
- 推奨アクション: 製品仕様（API レート制限の詳細）について直接ご回答ください
- KB ギャップ: API レート制限に関する記事が KB に存在しない可能性があります
```

**BAD例**:
```
引き継ぎサマリー:
「API に関する問い合わせです。確認してください。」
→ オペレーターが何を調べれば良いかわからず、顧客に再度問い合わせ内容を確認
→ 顧客にとって「同じことを何度も説明させられる」体験
→ CSAT 低下
```

### エラー 3: Q3 リペアパス後の再エスカレーション（ループ制御違反）

**症状**: Q3 → R4 → Q1〜Q3 → Q3 で再度「即エスカレーション判定」→ R4 へ無限ループ。  
**原因**: リペアは最大1回というループ制御が実装されていない。  
**対処**: R4 に `repair_path_count` カウンターを設ける。`repair_path_count = 1` で R4 に遷移した場合、次の Q3 判定では強制エスカレーション（Q4 で処理）とし、R4 に再遷移しない。`core-workflow.md` のループ制御ルールに準拠する。
