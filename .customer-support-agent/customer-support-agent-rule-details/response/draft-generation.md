# R3: Draft Generation — Phase Rule

## Purpose

R2 の KB 検索結果をもとに、メール回答ドラフトを生成する。KB 引用ルールを厳守しながら、顧客セグメント・感情シグナル・KB ルートに応じた適切な構造でドラフトを作成する。クレーム対応中セグメントには専用の5段階構造を適用する。CP-2 でドラフトのレビューを求める。

**Classification**: ALWAYS EXECUTE（即時回答モードおよび「確認の上回答」モード）
**Approval Gate**: あり（CP-2: 回答ドラフトのレビュー）
**参照元**: `core-workflow.md` R3 セクション

---

## Prerequisites

- R2 が完了しており、`knowledge_retrieval` フィールドが確定していること
- `kb_route` が `"immediate"` または `"will_confirm"` であること（`"escalation"` は R4 へルーティング済み）
- `common/output-structure-patterns.md` を読み込み済み（メール・テンプレートの構造参照元）
- `common/tone-guidelines.md` を読み込み済み（セグメント×感情のトーンマトリクス参照元）
- `common/content-validation.md` を読み込み済み（KB引用フォーマットの参照元）

---

## Execution Steps

### Step 1: ドラフトモードの確認

R2 の `kb_route` に基づき、ドラフト生成モードを確定する。

| kb_route | ドラフトモード | 使用するテンプレート |
|---------|------------|----------------|
| `immediate` | 即時回答ドラフト | 標準回答構造 or クレーム対応構造 |
| `will_confirm` | 「確認の上回答」ドラフト | 確認テンプレート（KB 情報を断定的に使用しない） |

### Step 2: 回答構造の選択

顧客セグメントと感情シグナルに基づき、適用する回答構造を選択する。

#### 標準回答構造（通常・一般顧客）

```
1. 挨拶 — 顧客名を使用したパーソナライズ（例: 「{顧客名} 様」）
2. 問題確認 — お問い合わせ内容の要約と理解確認
3. 解決策・回答 — KB引用に基づく手順または説明
4. 次のステップ — 顧客が次にとるべきアクション
5. 署名 — サービス名・問い合わせ窓口情報
```

#### クレーム対応構造（Complaint-in-progress セグメント、または frustrated/angry 感情シグナル）

v2 必須。以下の5段階構造をドラフト段階で実装する（Q3 で再確認される）。

```
1. 共感パラグラフ — 顧客の感情を受け止める表現（「ご不便をおかけして大変申し訳ございません」等）
2. 過去対応経緯の確認 — R2 で取得した過去解決事例を参照し、「前回〇〇の対応をさせていただいた件」等で文脈を共有
3. 謝罪 — 発生した問題に対する明示的な謝罪（「この度は〇〇により多大なご迷惑をおかけしました」等）
4. 解決策 — KB引用に基づく具体的な解決策または次のアクション
5. 次のステップ — フォローアップの約束（「〇〇日以内に進捗をご連絡します」等）
```

**VIP セグメントへの追加対応**:
- 標準・クレーム構造に関わらず、本文の敬語レベルを最上位に設定する
- 「優先対応させていただきます」等の優先性を示す表現を加える
- 担当者名（個人名）を署名に含める

#### 「確認の上回答」テンプレート（kb_route = will_confirm）

```
1. 挨拶
2. 問題確認
3. 確認の宣言 — 「ご質問の件につきまして、担当部門にて確認の上、改めてご回答申し上げます」
4. 回答期限の明示 — SLA に基づく期限（例: 「〇〇営業日以内にご連絡いたします」）
5. 暫定対処（あれば）— R2 の retrieved_articles から参考情報として提示可能な内容（断定的表現は禁止）
6. 次のステップ
7. 署名
```

### Step 3: KB 引用ルール（必須）

**ルール 1: KB 記事 URL の必須引用**

即時回答モードのドラフトには、使用した KB 記事の URL を必ず含める。

```
形式: 「詳細は以下のサポートページをご確認ください: {KB記事URL}」
```

**ルール 2: KB 外情報の生成禁止**

`retrieved_articles` に含まれない情報（手順・機能名・価格・仕様等）を回答に含めてはならない。不確かな情報は「確認の上回答」テンプレートに切り替える。

**ルール 3: 鮮度警告記事の取り扱い**

`freshness_warning: true` の KB 記事を使用する場合:
- 断定的な手順説明を避け、「現在の仕様については確認の上ご案内します」等の留保表現を加える
- Q2（Content Review）で強化チェックが入ることをドラフトメタデータに記録する

**ルール 4: 複数ステップ手順の形式**

操作手順が 3 ステップ以上の場合は番号付きリストを使用する。

```
例:
1. 設定画面（右上のアイコン）をクリックします
2. 「アカウント設定」タブを選択します
3. メールアドレス欄に新しいアドレスを入力します
```

### Step 4: パーソナライズ

以下の情報を使ってドラフトをパーソナライズする。

| 情報源 | 適用箇所 |
|------|---------|
| T1: 顧客名 | 挨拶（「{顧客名} 様」） |
| T1: プラン名 | 解決策（「{プラン} でご利用可能な機能として」等） |
| R2: 過去対応経緯 | クレーム対応構造の「過去対応経緯の確認」 |
| T3: SLA 期限 | 「確認の上回答」テンプレートの回答期限 |

### Step 5: ドラフトのメタデータ記録

```yaml
draft_generation:
  draft_mode: "{immediate | will_confirm}"
  response_structure: "{standard | complaint | vip_complaint | will_confirm}"
  kb_articles_cited:
    - article_id: "{KB記事ID}"
      url: "{記事URL}"
      freshness_warning: "{true | false}"
  personalization_applied: "{true | false}"
  complaint_structure_applied: "{true | false}"
  draft_content: "{生成したドラフト全文}"

audit_log:
  - timestamp: "{ISO 8601}"
    stage: "R3"
    action: "draft_generated"
    details: "mode: {draft_mode}, structure: {response_structure}, kb_cited: {件数}"
```

## Question Generation（CP-2）

R3 完了後、生成したドラフトを提示しユーザー確認を求める（CP-2）。

**CP-2 提示テンプレート**:

```
### REVIEW REQUIRED
回答ドラフトを生成しました。内容を確認してください。

**ドラフト情報**
- モード: {即時回答 | 確認の上回答}
- 回答構造: {標準 | クレーム対応 | VIP | 確認テンプレート}
- 使用 KB 記事: {記事タイトルリスト}
- 鮮度警告: {あり（Q2 で強化チェックされます） | なし}

---
**ドラフト本文**

件名: Re: {元件名}

{ドラフト本文全文}

---

### WHAT'S NEXT
**A) 修正依頼** — ドラフトを修正する
**B) Q1（PII Scan）へ続行** — 品質チェックへ進む
```

## Error Handling

### エラー 1: KB 未ヒットなのに推測で手順を案内（Pitfall: ハルシネーション回答）

**症状**: `retrieved_articles` が空または低スコアなのに、「おそらくこういう手順です」と推測回答を生成。  
**原因**: R2 から `kb_route: "escalation"` で渡されるべきところが `"will_confirm"` になっており、R3 で KB 情報なしのまま回答を生成する。  
**対処**: R3 は `kb_route` を必ず確認し、`"escalation"` の場合はドラフト生成を中断して R4 へ転送する。`"will_confirm"` であっても `retrieved_articles` が空の場合は R4 へルーティングする。

**GOOD例**:
```
kb_route: "will_confirm" / retrieved_articles: 2件（スコア 0.45, 0.42）
→ 確認テンプレートを使用
→ 「担当部門にて確認の上、2営業日以内にご回答します」
→ 2件の記事は「現在の仕様については確認します」等の留保表現で参考提示
→ KB 外情報は一切生成しない
```

**BAD例**:
```
kb_route: "will_confirm" / retrieved_articles: 0件
→ R3 でドラフト生成を試みる
→「設定 > アカウント管理 > セキュリティ設定からご変更いただけます」（KB になし）
→ ハルシネーション手順を顧客に送信
```

### エラー 2: クレーム対応中顧客に標準構造を適用（Pitfall: トーンミスマッチ）

**症状**: `segment: Complaint-in-progress` の顧客に対し、共感・謝罪なしの事務的な手順説明メールを送付。  
**原因**: Step 2 でセグメントチェックが未実装。標準回答構造がデフォルトで適用される。  
**対処**: Step 2 の構造選択ロジックで `segment = Complaint-in-progress` または `emotion_signal IN (frustrated, angry)` を最初にチェックし、クレーム対応構造を適用する。

**GOOD例**:
```
顧客: segment = Complaint-in-progress / emotion_signal = frustrated
→ クレーム対応構造を選択
→ 1. 共感「この度は長期にわたりご不便をおかけし、誠に申し訳ございません」
→ 2. 過去経緯「先日〇〇の件でご連絡をいただいておりました件」
→ 3. 謝罪「再度このような事態を招いてしまったことをお詫び申し上げます」
→ 4. 解決策「本日中に〇〇の設定を修正いたします」
→ 5. 次のステップ「1営業日以内に完了のご連絡をさせていただきます」
```

**BAD例**:
```
顧客: segment = Complaint-in-progress / emotion_signal = frustrated
→ 標準構造を適用
→「お問い合わせありがとうございます。以下の手順をお試しください:...」
→ 顧客: 「謝罪もなく手順を送り付けてきた」とさらに激化
→ CSAT 1（最低評価）
```

### エラー 3: KB 記事 URL を引用せずドラフトを生成（Pitfall: ハルシネーション回答）

**症状**: 即時回答ドラフトに解決手順が含まれているが、KB 記事 URL が記載されていない。  
**原因**: Step 3 の KB 引用ルール（ルール 1）が実装されていない。  
**対処**: 即時回答ドラフトには必ず使用 KB 記事の URL を本文末尾に含める。Q2（Content Review）で KB 引用の存在を確認する。引用なしの場合は Q2 で自動的にフラグを立てて修正を要求する。

### エラー 4: VIP 顧客への敬語レベルが標準のまま（Pitfall: トーンミスマッチ）

**症状**: VIP 顧客への回答が一般顧客向けと同一の敬語レベル。  
**原因**: `segment = VIP` の場合のトーン強化ロジックが未実装。  
**対処**: `segment = VIP` の場合は `common/tone-guidelines.md` の VIP 向けトーン指針を参照し、敬語レベルを最上位に設定する。署名に担当者個人名を含め、「優先対応させていただきます」等の表現を必ず加える。

**GOOD例**:
```
顧客: segment = VIP / emotion_signal = normal
→ VIP 回答構造
→「山田様、いつも大変お世話になっております。今回のご連絡、誠にありがとうございます。
  本件につきましては、優先対応させていただきます。」
→ 署名: 「カスタマーサポート 田中 太郎（担当）」
```

**BAD例**:
```
顧客: segment = VIP / emotion_signal = normal
→ 標準トーンで回答
→「お問い合わせありがとうございます。以下をご確認ください。」
→ VIP 顧客が「普通の対応しか受けられない」とチャーンリスク上昇
```
